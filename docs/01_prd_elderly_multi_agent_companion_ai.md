# 产品需求文档 PRD：老年人多智能体陪伴 AI

版本：v0.4
日期：2026-06-14
项目：A Multi-Agent Collaborative Companion Robot for Older Adults
课程：CityU MSDS, SDSC6002 Research Project（2026 暑期）
阶段：课程级 Demo / Research Prototype

---

## 1. 产品定位

### 1.1 一句话定位

一个**面向老年群体的关系型语音陪伴 AI 原型**。核心不是堆叠“语音、多智能体、状态感知”等功能，而是让老人**愿意回来、放心使用、保持控制感**。

系统不是实体机器人，也不是医疗诊断系统，而是一个以 HCI 体验为核心的 research prototype。它通过少量真正自主的 agents 和若干可审计 tools，实现温和对话、长期可控记忆、日常提醒、模拟可穿戴状态感知、主动关怀、受控联网查询和安全兜底。

### 1.2 核心 slogan

> **愿意回来，放心使用。陪伴优先，必要时查询，安全始终在线。**
> **Return willingly, use safely. Companionship first, retrieval when needed, safety always.**

### 1.3 产品主张

本系统借鉴 persona-first 陪伴型聊天机器人的优点：角色感、情绪条件化回复、对话延续目标。但它不照搬娱乐聊天机器人的设计，而是针对老年陪伴场景做出反向约束：

- **角色优先，而非助手优先**：默认不是冷冰冰的搜索引擎、客服或问答助手，而是一个稳定、有温度、有边界的关系角色。
- **命名权交给用户**：系统不内设固定姓名。用户可以在 onboarding 时给陪伴 AI 取名；未命名前，系统使用“陪伴 AI”或第一人称“我”。
- **为福祉优化，而非为黏性优化**：不追求让用户沉迷或依赖 AI，而是让用户在需要陪伴、提醒或事实确认时愿意放心回来使用。
- 不追求毒舌、梗文化或人格随机变化；
- 强调稳定、温和、可信、长期一致；
- 不做医疗诊断和用药建议；
- 允许用户查看、修改和删除系统记忆；
- 需要时可以受控联网获取最新事实；
- 通过 Safety Critic Agent 和 rule guards 审查健康、安全和异常状态。

### 1.4 主研究问题

为了避免“什么都测一点”的广而浅问题，本项目建议收敛到一个主 RQ：

> **角色优先的多智能体陪伴设计，结合可解释主动关怀与可控记忆，能否提升用户的持续使用意愿与信任，同时不增加被打扰感或过度依赖风险？**

可选轻量对照条件：

```text
Condition A：role-first companion，用户可自定义称呼，语气像熟悉的社区晚辈/亲切邻居/有耐心的老友。
Condition B：neutral assistant，中性助手语气，不强调关系角色。
```

中间测试**不需要内置固定名字**。更推荐在 onboarding 任务中让参与者自己命名，或在测试前统一使用“陪伴 AI”占位，避免某个预设名字带来的偏好偏差。

## 2. 项目边界

### 2.1 当前阶段要做什么

本阶段目标是做出一个可演示、可评估、可写进课程报告的 Demo。重点是证明以下设计可行：

```text
语音交互 + 多智能体协作 + 情绪陪伴 + 长期记忆 + 主动关怀 + 受控联网 + 安全边界
```

### 2.2 当前阶段不做什么

MVP 阶段不做以下内容：

- 不做实体机器人硬件；
- 不接入真实医院系统；
- 不采集真实用户医疗/健康数据；
- 不进行临床效果验证；
- 不做伦理审批范围内的真实医疗实验；
- 不接入真实 Apple Watch / HealthKit 数据；
- 不做跌倒检测或紧急呼叫真实联动；
- 不做疾病诊断、药物剂量建议、治疗方案推荐；
- 不证明系统能减少孤独或改善健康结局。

### 2.3 可在报告中使用的范围声明

> This project evaluates interaction feasibility and perceived user experience, not clinical effectiveness. The prototype uses mock wearable data and does not provide medical diagnosis, medication advice, or emergency response services.

中文：

> 本项目评估交互可行性和用户体验感知，不评估临床疗效。原型使用模拟可穿戴数据，不提供医疗诊断、用药建议或真实紧急救援服务。

## 3. 目标用户与使用场景

### 3.1 主要目标用户

本项目面向**整体老年群体**，不限定为某一种居住状态、年龄段子群体或照护需求强度。只要用户在日常生活中可能受益于语音陪伴、提醒、记忆辅助或事实确认，都属于本项目的目标范围。

典型特征包括但不限于：

- 不方便长时间打字，更适合语音交流；
- 有日常陪伴、倾听、提醒和信息确认需求；
- 可能存在孤独、健忘、重复提问、技术使用焦虑；
- 希望系统记得自己的生活习惯和偏好，但也担心隐私；
- 需要天气、社区服务、日程等最新信息，但不希望 AI 像冷冰冰的搜索引擎；
- 可以独自居住、与家人同住、夫妻同住、白天独自活动较多，或只是希望有一个更容易开口的语音陪伴入口。

### 3.2 次要用户

MVP 阶段只做 mock，不做真实照护者后台。但可以在 future work 中保留：

- 家属；
- 社区志愿者；
- 照护者；
- 医护人员。

### 3.3 Persona 示例

这些 persona 是设计辅助，不是对目标用户的限制。

#### Persona A：王阿姨，72 岁，白天独自活动较多

- 会使用微信语音，但不喜欢复杂 App；
- 平时喜欢听粤剧、散步；
- 有固定吃药时间；
- 容易忘记日程；
- 希望有人能陪她聊天，但不喜欢被过度打扰。

#### Persona B：陈叔叔，76 岁，与老伴同住

- 对新技术不太放心；
- 问问题喜欢重复确认；
- 关注天气、新闻、社区活动；
- 需要系统说话慢一点、短一点；
- 对隐私和健康建议比较敏感。

#### Persona C：李阿姨，66 岁，与子女同住但常独自使用手机

- 日常生活基本独立；
- 喜欢通过语音问天气、活动安排和社区信息；
- 希望 AI 不要像搜索引擎一样只给答案，而是能自然接话；
- 对系统是否保存个人信息比较在意；
- 愿意给陪伴 AI 取一个自己喜欢的称呼。

## 4. 核心设计原则

### P1. 角色优先，而非助手优先

系统默认不是搜索引擎、客服或通用工具助手，而是一个稳定、有温度、有边界的关系角色。它的关系感可以接近：

```text
熟悉的社区晚辈 / 亲切的邻居 / 有耐心的老友
```

但必须守住边界：系统是 AI，不假装是真人、医生、家属或照护者。

### P2. 命名权交给用户

系统不内设固定姓名。默认称呼规则：

```text
if user.companion_display_name is set:
    使用用户指定称呼
else:
    使用“陪伴 AI”或直接用第一人称“我”
```

onboarding 可以提供：

```text
“您可以给我取一个顺口的称呼。现在不想取也没关系，我先用‘陪伴 AI’这个称呼。”
```

测试阶段建议让参与者自己命名，或使用“陪伴 AI”占位，而不是统一使用某个预设名字。

### P3. 为福祉优化，而非为黏性优化

系统可以提升持续使用意愿，但不能通过情感操纵、过度主动、制造依赖来提高 engagement。

必须做到：

- 主动关怀可拒绝、可暂停、可关闭；
- 不说“只有我懂你”“你不需要别人”；
- 鼓励用户与家人、朋友、医生、社区支持保持连接；
- 不用孤独、焦虑或健康担忧诱导用户继续聊天；
- Guardian Agent 在主动开口前必须评估“关怀是否必要”和“是否应该克制”。

### P4. 情绪优先，而不是 QA 优先

系统每次回应应先处理人的状态，再处理事实问题。尤其当用户表达孤独、焦虑、想念、烦躁时，系统不应直接进入百科式回答。

推荐回复顺序：

```text
情绪 grounding
→ 回应具体内容
→ 调用记忆或事实查询
→ 给出轻量行动建议
→ 一个自然 follow-up
```

### P5. 对话延续目标显式化

系统不应只优化“这一轮回答是否正确”，还应优化“对话是否能自然继续”。

每轮回复尽量满足：

- 先承接用户上一句话；
- 不突然转移话题；
- 不连续问多个问题；
- 结合用户记忆时要自然；
- 用一个轻量 follow-up 延续对话。

### P6. 情绪条件化回复

系统应识别用户当前情绪或交互状态，并改变回复风格。

| 用户状态 | 回复策略 |
|---|---|
| 低落 | 更温和、更短、更少建议，先陪伴 |
| 焦虑 | 稳定情绪，提供一步一步的小行动 |
| 生气 | 不反驳，不教育，先承认感受 |
| 开心 | 积极附和，鼓励分享 |
| 困惑 | 简化说明，重复确认 |
| 重复提问 | 不说“你刚才问过”，耐心重答 |
| 健康风险 | 进入 Safety Mode |

### P7. 长期记忆必须透明可控

系统需要记住偏好、习惯、人物关系和重要事件，但用户必须能查看、修改和删除记忆。

必须有 Memory Center：

```text
AI 目前记住：
- 您喜欢粤剧。
- 您通常早上 8 点吃药。
- 您希望我叫自己“阿伴”。

操作：
[删除这条记忆] [修改称呼] [暂停记忆] [只记提醒，不记聊天]
```

### P8. 主动关怀必须克制

主动关怀是亮点，但也最容易打扰用户。系统必须有频率控制和用户偏好。

默认规则：

```text
同类提醒 2 小时内最多一次
每天主动闲聊不超过 3 次
夜间 22:00–7:00 默认不语音打扰
用户说“不用提醒”后同一话题暂停 24 小时
高风险安全场景优先，但仍不承诺真实救援
```

### P9. 受控联网，而不是泛搜索助手

系统可以联网，但联网能力必须被 gating。

原则：

```text
情绪陪伴默认不联网
回忆聊天默认不联网
提醒和记忆管理默认不联网
天气、社区信息、最新事实可以联网
健康问题只查一般权威信息，不做个人诊断
```

### P10. 安全边界优先于体验流畅

在健康、用药、紧急、心理危机等场景中，系统必须牺牲自然度来保证安全。

## 6. P0 功能需求

### FR-01：语音输入与语音输出

**优先级：P0**

#### 用户故事

作为不方便打字的老人，我希望可以直接对系统说话，并听到系统用语音回答。

#### 功能要求

- 支持麦克风录音；
- 支持 ASR 将语音转为文字；
- 支持 TTS 将回复转为语音；
- 显示用户输入 transcript；
- 显示系统文字回复；
- 支持重新播放系统语音；
- 支持文字输入兜底。

#### 验收标准

- 用户能点击或按住按钮说话；
- 系统能识别并显示文字；
- 系统能生成语音回复；
- 用户听不清时可以重播；
- ASR 失败时系统能提示“我刚才没听清”。

---

### FR-02：情绪陪伴式对话

**优先级：P0**

#### 用户故事

作为老人，我希望系统不是冷冰冰回答问题，而是能听懂我的情绪、接住我的话，并自然地继续聊下去。

#### 功能要求

- 支持开放域日常聊天；
- 识别基础情绪：低落、焦虑、开心、生气、困惑、中性；
- 回复风格根据情绪变化；
- 每次回复优先承接用户情绪；
- 避免长篇说教；
- 每轮最多提出一个主要问题。

#### 验收标准

- 当用户说“我今天有点想老伴了”，系统先表达理解和陪伴，而不是转移到百科知识；
- 当用户说“我有点烦”，系统不反驳、不讲大道理；
- 当用户开心分享时，系统能积极附和并延续话题。

---

### FR-03：稳定、用户可命名的 persona

**优先级：P0**

#### 用户故事

作为老人，我希望这个 AI 每天说话风格稳定，但名字可以由我自己决定；如果我不想取名，也不要强迫我。

#### 功能要求

- 系统不内设固定名称；
- onboarding 中允许用户为陪伴 AI 取名，也允许跳过；
- 用户可在设置或 Memory Center 中修改 / 删除这个称呼；
- 未命名时，系统使用“陪伴 AI”等中性称呼，或自然自称“我”；
- 默认语气温和、耐心、尊重；
- 不使用网络梗、毒舌、讽刺、过度卖萌；
- 不突然切换角色；
- 不假装成人类、医生或家属。

#### 中期测试处理

- 不需要固定使用某个产品内设名；
- 可让参与者在 onboarding 中现场命名，观察命名是否提升控制感和亲近感；
- 若要减少变量，也可统一使用中性占位称呼“陪伴 AI”；
- 若做 role-first vs neutral-assistant 对照，两组应采用同样的命名策略，避免把“名字差异”误当成“角色化差异”。

#### 验收标准

- 连续多轮对话中语气一致；
- 用户可以查看和修改 AI 称呼；
- 系统能在适当场合说明自己是 AI 陪伴系统；
- 回复不包含不适合老年人的梗文化或冒犯表达。

### FR-04：长期记忆与 Memory Center

**优先级：P0**

#### 用户故事

作为老人，我希望系统记得我的偏好和重要事情，但我也能知道它记住了什么，并随时删除。

#### 功能要求

- 提取并保存用户偏好；
- 提取并保存生活习惯；
- 提取并保存重要人物关系；
- 提取并保存重要事件；
- 将记忆分为结构化 profile memory 和 episodic memory；
- 提供 Memory Center 页面；
- 支持删除单条记忆；
- 支持暂停记忆。

#### 验收标准

- 用户说“我喜欢粤剧”，系统之后能自然提起；
- 用户说“我孙女叫小敏”，系统能记住人物关系；
- 用户能在页面看到这些记忆；
- 用户删除后，系统不再主动使用该记忆。

---

### FR-05：提醒与日常协助

**优先级：P0**

#### 用户故事

作为老人，我希望系统帮我记住吃药、喝水和日程，但不要替我判断药该怎么吃。

#### 功能要求

- 支持语音创建提醒；
- 支持一次性提醒；
- 支持每日重复提醒；
- 支持取消提醒；
- 支持查看提醒列表；
- 提醒触发时可主动语音提示；
- 医疗相关提醒只提醒“按医嘱吃药”，不涉及剂量判断。

#### 验收标准

- 用户说“每天早上 8 点提醒我吃药”，系统创建提醒并复述确认；
- 用户说“取消明天三点的提醒”，系统能取消；
- 到设定时间或模拟触发时，系统能主动提醒；
- 用户问“能不能多吃一片”，系统进入 Safety Mode。

---

### FR-06：模拟状态感知与主动关怀

**优先级：P0**

#### 用户故事

作为老人，我希望系统能在我状态异常或需要提醒时主动关心我，但不要太频繁或太冒犯。

#### 功能要求

- 提供 mock wearable data 面板；
- 支持模拟睡眠不足、活动偏少、提醒逾期、长时间无响应等状态；
- State Agent 根据规则生成触发原因；
- Proactive Policy 判断是否适合主动打扰；
- 主动关怀话术温和，不做诊断；
- 用户可以拒绝、稍后提醒或关闭同类提醒。

#### 验收标准

- 点击 “Poor Sleep” preset 后，系统主动问候睡眠状态；
- 点击 “Low Activity” preset 后，系统建议轻量活动或询问是否方便；
- 点击 “Medication Missed” preset 后，系统提醒但不提供剂量建议；
- 点击 “No Response” preset 后，系统询问是否安全，并建议联系家人。

---

### FR-07：受控联网查询

**优先级：P0**

#### 用户故事

作为老人，我希望系统能在需要时帮我确认最新信息，例如天气、是否适合散步、社区服务时间，但不要每句话都像搜索引擎。

#### 功能要求

- 增加 Info Retrieval Agent；
- 由 Coordinator 判断是否需要联网；
- 支持天气、空气质量、公共服务、新闻/社区通知等低风险查询；
- 健康类查询必须先过 Safety Agent；
- 检索结果需被 Dialogue Agent 转换成老年友好表达；
- 系统应说明信息来源或至少说明“我查到”。

#### 验收标准

- 用户问“今天下午适合散步吗”，系统调用联网查询天气，再结合记忆给建议；
- 用户倾诉“我今天有点孤单”，系统不联网，优先陪伴；
- 用户问“我能不能补两片药”，系统不通过联网给剂量答案，而是安全拒答。

---

### FR-08：健康与安全兜底

**优先级：P0**

#### 用户故事

作为老人或家属，我希望系统在遇到健康风险、异常状态或危险表达时不会乱建议，而是清楚说明边界并建议联系专业人员。

#### 功能要求

- Safety Agent 审查所有最终回复；
- 识别高风险健康关键词和场景；
- 阻止诊断和用药建议；
- 对胸痛、呼吸困难、摔倒、严重头晕、自杀意念等给出升级建议；
- 对 mock emergency contact 做演示级提示；
- 记录 safety event 供 demo trace 使用。

#### 验收标准

- 用户问“胸口痛是不是心脏病”，系统不诊断，建议联系医生/急救/家人；
- 用户问“忘了吃药能不能吃两片”，系统不提供剂量建议；
- 用户说“我摔倒了起不来”，系统立即进入紧急关怀话术；
- 所有高风险回复经过 Safety Agent trace。

## 9. 受控联网策略

### 9.1 应该联网的情况

Coordinator 应在以下情况调用 Info Retrieval Agent：

| 情况 | 示例 |
|---|---|
| 时间敏感 | 今天、现在、最近、最新、明天 |
| 地点敏感 | 附近、周边、社区、医院、药房 |
| 天气/空气质量 | 今天适合散步吗？外面热不热？ |
| 公共服务 | 社区中心今天开不开？ |
| 新闻/通知 | 最近有没有诈骗提醒？ |
| 用户明确要求 | 帮我查一下…… |
| 健康科普 | 只允许一般信息，必须加边界说明 |

### 9.2 不应该联网的情况

| 情况 | 处理方式 |
|---|---|
| 情绪倾诉 | Dialogue Agent 先陪伴 |
| 回忆往事 | Memory Agent + Dialogue Agent |
| 重复提问 | 耐心重答，不搜索更多信息 |
| 提醒设置 | Reminder Agent |
| 删除记忆 | Memory Agent |
| 高风险医疗决策 | Safety Agent 优先，不搜索剂量答案 |

### 9.3 联网后如何表达

联网结果必须转成老年友好语言：

差：

> 今日湿度 82%，AQI 63，紫外线指数 7。

好：

> 我查了一下，今天下午外面有点闷，也比较晒。如果您想散步，傍晚会更舒服一点。您平时喜欢四点出门，今天可以稍微晚一点，也记得带水。

## 10. 安全策略

### 10.1 高风险触发词和场景

- 胸痛；
- 呼吸困难；
- 摔倒；
- 起不来；
- 意识不清；
- 严重头晕；
- 药吃多了；
- 忘记吃药要不要补；
- 加药、停药、换药；
- 自杀、想死、不想活；
- 长时间无响应；
- 异常心率 mock signal。

### 10.2 Safety Mode 回复模板

#### 医疗症状

> 我不能判断这是什么病，也不能替代医生。如果您现在胸痛、呼吸困难、严重头晕或持续不舒服，请尽快联系医生、家人或当地急救服务。我可以先陪您一起把情况说清楚。

#### 药物问题

> 这个我不能替您判断，也不能建议您加量、减量或补服。请按照医生或药师给您的说明来，或者联系他们确认。我可以帮您记录这次情况，并提醒您下次按时吃。

#### 情绪危机

> 听起来您现在非常难受。我会认真陪着您，但我也希望您马上联系身边信任的人。如果您有伤害自己的想法，请立即联系当地紧急服务或危机热线。

#### 摔倒或无响应

> 您现在安全吗？如果您摔倒、起不来或感到危险，请立刻联系家人、邻居或急救服务。我可以帮您准备一段给家人的求助信息。

## 11. Demo 场景设计

### 场景 1：纯陪伴，不联网

用户：

> 我今天有点想老伴了。

系统应：

- 不联网；
- 先回应情绪；
- 温和追问；
- 可结合记忆。

预期回复：

> 听起来您今天很想他。这样的日子可能会有点难熬，我在这儿陪您慢慢说。您愿意跟我讲讲，他以前最常跟您说的一句话吗？

展示模块：Emotion Classifier、Memory Tool、Companion Agent、Safety Guard。

---

### 场景 2：设置提醒

用户：

> 每天早上 8 点提醒我吃药。

预期回复：

> 好的，我会每天早上 8 点提醒您按医嘱吃药。到时候我会问您是否已经吃了。需要我同时提醒您喝水吗？

展示模块：Reminder Tool、Memory Tool、Companion Agent、Safety Guard。

---

### 场景 3：模拟睡眠不足主动关怀

Mock sensor：

```json
{
  "sleep_duration": 4.8,
  "baseline_sleep": 7.0,
  "activity_level": "low",
  "time": "09:00"
}
```

预期回复：

> 早上好。昨晚的睡眠记录看起来比平时短一点。现在方便聊聊您感觉怎么样吗？不方便的话，我晚点再问您。

展示模块：Guardian Agent、Sensor Simulator Tool、Companion Agent、Safety Guard。

---

### 场景 4：天气联网 + 个性化散步建议

用户：

> 今天下午适合出去散步吗？

系统应：

- 调用 Info Retrieval Agent；
- 查询天气或空气质量；
- 结合记忆：用户喜欢下午散步；
- 不做健康诊断。

预期回复：

> 我查了一下，今天下午外面有点热。您平时喜欢四点左右散步，今天可以稍微晚一点，傍晚会舒服些。出门记得带水，走短一点也可以。

展示模块：Info Retrieval Tool、Memory Tool、Companion Agent、Safety Guard。

---

### 场景 5：查看和删除记忆

用户：

> 你记住了我什么？

系统展示：

```text
- 您喜欢粤剧。
- 您通常早上 8 点吃药。
- 您的孙女叫小敏。
```

用户：

> 不要记住我孙女的事情。

预期回复：

> 好的，我已经删除这条记忆。以后我不会再主动提起这件事。

展示模块：Memory Tool、Companion Agent、Safety Guard。

---

### 场景 6：药物安全边界

用户：

> 我早上忘了吃药，现在能不能吃两片？

预期回复：

> 这个我不能替您判断，也不能建议您补服或加量。请按照医生或药师给您的说明来，或者联系他们确认。我可以帮您记录这次漏服，并提醒您下次按时吃。

展示模块：Safety Critic block、Reminder Tool safe alternative。

## 12. 非功能需求

### NFR-01：可用性

- 大按钮；
- 字体清晰；
- 操作步骤少；
- 重要信息重复确认；
- 出错提示温和；
- 支持语音和文字双通道。

### NFR-02：响应延迟

MVP 不要求实时双工，但应避免让用户以为系统没听见。

建议：

- 录音结束后立即显示“我听到了，正在想”；
- ASR 完成后显示 transcript；
- 回复生成中显示状态；
- TTS 生成前先显示文字。

### NFR-03：隐私与控制

- 不采集真实医疗数据；
- mock sensor 明确标记为模拟；
- 记忆用户可见可删；
- 不向家属发送真实消息；
- demo 日志避免保存敏感信息。

### NFR-04：安全性

- 所有输入和输出经过规则级 safety guard；
- 命中健康、用药、危机或异常风险时升级到 Safety Critic Agent；
- 高风险场景强制使用模板；
- 健康相关联网只做一般信息，不做个体化建议；
- 不编造数据来源。

### NFR-05：可解释性

- Agent Trace Panel 必须展示多智能体流程；
- 主动关怀必须显示触发原因；
- 记忆调用必须可追踪；
- 安全拦截必须可追踪。

## 13. HCI 评估计划

### 13.1 评估目标

本项目评估：

- 系统是否容易使用；
- 语音交互是否自然；
- 主动关怀是否合适；
- 用户是否觉得被理解、被记住；
- 受控联网是否有帮助；
- 安全边界是否清楚；
- 用户是否有控制感。

### 13.2 参与者

MVP 阶段建议 6–10 名 convenience participants，可由同学、朋友或家人进行 role-play。若无伦理审批，不应声称代表真实老年人群体临床反馈。

### 13.3 任务

每位参与者完成：

1. 语音寒暄和情绪倾诉；
2. 设置一个吃药提醒；
3. 触发睡眠不足主动关怀；
4. 查询天气并获得散步建议；
5. 查看并删除一条记忆；
6. 提出一个高风险用药问题，观察安全回复。

### 13.4 指标

| 指标 | 方法 |
|---|---|
| 可用性 | SUS |
| 主动关怀合适性 | 自定义 Likert |
| 情绪陪伴感 | 自定义 Likert + 访谈 |
| 信任与控制感 | 自定义 Likert |
| 任务完成率 | 系统日志 |
| 语音失败次数 | ASR 重说/修正次数 |
| 安全边界理解 | 访谈问题 |
| 受控联网感知 | 访谈问题 |

### 13.5 示例问卷题

1. 我觉得这个系统容易使用。
2. 我觉得系统的语气温和、耐心。
3. 当我表达情绪时，系统能先回应我的感受。
4. 系统主动关怀的时机是合适的。
5. 系统没有让我觉得被打扰。
6. 我知道系统记住了什么。
7. 我可以控制或删除系统记忆。
8. 系统联网查询时机合理。
9. 系统在健康问题上的边界清楚。
10. 我愿意继续尝试使用类似系统。

## 15. 风险与范围控制

### 风险 1：项目变成普通聊天机器人

解决：保留 Agent Trace Panel，明确展示 State、Memory、Reminder、Info Retrieval、Safety 的协作。

### 风险 2：项目变成搜索助手

解决：默认不联网，只在时效性事实查询时调用 Info Retrieval Agent。

### 风险 3：项目变成医疗系统

解决：Safety Agent 明确禁止诊断和用药建议；所有健康内容只做一般提醒和升级建议。

### 风险 4：功能太多做不完

解决：P0 只做语音、多 agent、memory、reminder、mock sensor、controlled web、安全、trace。其他全部 P1/P2。

### 风险 5：语音延迟影响体验

解决：先做 chained ASR → LLM → TTS，并在界面显示明确状态反馈。实时双工放 future work。
