# SDSC6002 老年人多智能体陪伴 AI：文档索引

版本：v0.4
日期：2026-06-14
项目名：A Multi-Agent Collaborative Companion Robot for Older Adults
当前阶段：课程级软件 Demo / HCI research prototype，不做真实医疗部署

---

## 一句话定位

本项目研究一个**面向老年群体的关系感知（relationship-aware）回忆陪伴 AI**。核心不是堆叠功能，而是围绕三个核心机制，让不同 AI 关系角色**动态编排、以社会线索自然引导老人进入回忆与自我表达**：

1. **关系编排（relationship orchestration）**：基于话题、人生经历、情绪状态、关系偏好与边界需求，动态决定谁说话、谁沉默、谁追问、谁总结、何时停止。
2. **社会线索引入话题（social cueing）**：让 2–3 个 AI 关系角色围绕一张老照片、一件旧物或一个人生话题先进行简短对话，形成轻量的多人闲聊场景，再邀请老人加入。
3. **多智能体角色交互（multi-agent role interaction）**：让不同关系角色——同龄共鸣、晚辈好奇、中年传承、回忆整理、边界守护——彼此协作，避免噪声（吵）、失真（假）、过载（乱）和越界。

情绪陪伴、可控记忆、日常提醒、模拟状态感知、主动关怀、受控联网查询和健康安全兜底是支撑这三个机制的**技术底座与支撑能力**，而不是研究主线本身。

> **关系优先，线索自然。让角色协作把老人温和地带入回忆，而不是不停追问。**
> **Relationship first, cueing natural. Let roles cooperate to gently invite recollection, not interrogate.**

### 主研究问题

> 老年回忆陪伴中，系统如何基于话题、人生经历、关系偏好和边界需求，动态编排不同 AI 关系角色，并通过多智能体短对话作为社会线索，更自然地引导老人进入回忆与自我表达？

### 三个子问题（对应三个核心机制）

- **RQ1 关系编排**：不同话题下哪些关系角色更适合出现？系统如何决定谁说话、谁沉默、谁总结、何时停止？
- **RQ2 社会线索引入话题**：agent-agent short cue（角色间简短对话）是否比直接提问更自然、更低压力、更容易启动回忆？
- **RQ3 多智能体角色交互**：同龄共鸣、晚辈好奇、中年传承、回忆整理、边界守护等角色如何协作，才能避免噪声和越界？

**边界问题作为伦理讨论（不是并列的第 4 个 RQ）**：当话题触及逝者、哀伤、隐私或依赖时，系统如何暂停、转向或婉拒。

### 核心贡献

- **关系角色分类（relationship-role taxonomy）**：同龄共鸣 / 晚辈好奇 / 中年传承 / 回忆整理 / 边界守护。
- **动态关系编排（dynamic relationship orchestration）**：按话题、经历、偏好和边界决定角色的出场、沉默、追问、总结与停止。
- **社会线索引入话题（agent-agent conversational cueing）**：把角色间的简短对话当作一种社会记忆线索。
- **角色交互规则**：共鸣 / 追问 / 总结 / 沉默 / 边界守护。
- **角色 / 话题 / 边界 trace**：让编排与越界处理可观察、可解释。

> **范围诚实**：当前代码是**技术底座与 Demo 骨架**，不是研究全部。下一阶段将在此之上叠加关系感知原型 / Wizard-of-Oz / 半自动编排层。尚未开展真实老人实验；真实 ASR/TTS 可选可用，真实 LLM 与真实联网检索为 provider 接口 / 未来工作，`DEMO_MODE` 默认仍为离线 fake/mock（无需密钥）。

## 命名原则

系统**不内设固定姓名**。陪伴角色的称呼权交给用户：

- onboarding 时可以让用户选择或自定义称呼；
- 用户未命名前，系统默认自称为“陪伴 AI”或直接用第一人称“我”；
- 测试阶段不需要固定使用某个名字，避免姓名偏好影响实验结果；
- 代码中使用 `companion_display_name` 保存用户自定义称呼，默认值为空；未命名时仅在 UI / prompt 层 fallback 为“陪伴 AI”。

这能同时保留“角色优先”的关系感，又避免把某个内置名字强加给所有老人。

---

## 目标用户范围

目标用户是**整体老年群体**，不限定为某一种居住状态、年龄段子群体或照护需求强度。独自居住、与家人同住、白天常独自活动、技术焦虑、需要日常提醒、社区活动较多等都只是典型使用场景。

MVP 评估时可以使用 role-play 或 convenience participants；若后续招募真实老年人，需要根据导师要求处理伦理与知情同意。

## 建议最小可演示场景

围绕三个核心机制展开，功能能力作为支撑：

1. **社会线索引入话题**：围绕一张老照片，2–3 个 AI 关系角色（如同龄共鸣、晚辈好奇）先简短对话，再邀请老人加入，而不是直接抛问题。
2. **关系编排**：不同话题下系统决定哪个角色出场、谁沉默、谁追问、谁总结、何时停止。
3. **边界守护**：当话题触及逝者、哀伤或隐私时，系统暂停、转向或婉拒（伦理讨论，非并列 RQ；绝不扮演逝者）。
4. 老人用语音聊天，系统先回应情绪，再自然追问（角色优先的对话风格）。
5. onboarding 时用户给陪伴 AI 取一个称呼；未命名时系统使用“陪伴 AI”。
6. （支撑能力）系统记住并复用偏好，例如喜欢粤剧、孙女周日来访；老人可查看并删除一条记忆——长期记忆在此作为支撑机制与边界机制，而非独立卖点。
7. （支撑能力）老人设置吃药或喝水提醒；系统基于 mock wearable data 主动问候。
8. （支撑能力）老人问天气或是否适合散步，系统受控联网查询。
9. （安全底线）老人问高风险健康问题，系统拒绝诊断和用药建议，并建议联系医生/家属/急救；不做真实急救呼叫/短信/派单。
