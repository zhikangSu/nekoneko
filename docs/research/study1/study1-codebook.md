# Study 1 Codebook

> Study 1（第一阶段设计访谈）的**定性分析编码本**。用来把 **pilot 试访谈**和后续正式 Study 1 的访谈笔记，转成可复用、可统计的编码（code）。
>
> **这是初步版本，会在 pilot 后再迭代。** pilot 的主要作用之一，就是检验这 10 类编码是否够用、是否有遗漏、子码是否需要拆分或合并（详见文末「§13 pilot 后如何修订本编码本」）。
>
> **配套文件（同目录，务必对照使用）：**
> - `topic-cards.md` —— 话题卡 T01–T12（话题偏好、敏感度）
> - `role-cards.md` —— 角色卡 R1–R8（关系角色、话题—角色矩阵规则）
> - `conversation-cue-scripts.md` —— 引入方式脚本 C1–C4，每组含 A 直接提问 / B 单角色铺垫 / C 多角色短对话
> - `memory-card-examples.md` —— 记忆卡片 M1–M5，及记忆边界规则
> - `interview-record-template.md` —— 访谈记录表（§4 就是「话题—关系角色矩阵」）
> - `study1-interview-guide-v2.md` —— 访谈五段流程与敏感话题原则
>
> **范围与伦理提示：** 本文件为研究准备材料，用于 pilot（1–2 人先导访谈）。尚未招募真实参与者、不含任何 PII，正式访谈前须与导师确认伦理审查与知情同意。安全底线：不做医疗诊断、不给用药剂量建议；涉及已故亲友（T09）时，系统与任何角色**绝不扮演逝者**。

## 0. 怎么用这份编码本

**它是什么：** 一套「贴标签」的规则。访谈结束后（或边听录音笔记边读记录表时），研究者把参与者说的每一段有意义的话，归到下面 10 类编码之一或几类，并尽量落到具体子码上。这样多位参与者、多次访谈的数据就能横向比较、汇总成设计依据。

**编码单位（coding unit）：** 以「一个有意义的表态或事件」为单位——可能是一句话、一次卡片选择、一次犹豫、一次喊停。同一段话可以同时打多个码（例如既是 `Topic preference` 又是 `Ethical boundary`）。

**证据要求：** 每个码后面尽量附一句**参与者原话（去标识）** + **来源**（访谈段落 A–E / T-code / R-code / C 组 / M-code）。原话是编码的凭据，也是后续报告里的引用来源。

**编码记法建议：** `类别 [子码] 来源 「原话」`。例如：
> `Relationship-role fit [T05→R1] C段 「聊老照片嘛，还是找个年纪差不多的一起唠更亲」`
> `Role rejection [R2:像被盘问] C段 「小孩子一句接一句地问，我招架不住」`
> `Cue preference [C>A] D段 「几个人先聊着，我插一句就行，没那么大压力」`

**10 类一览：**

| 编码类别 | 一句话抓什么 | 主要来源 | 关联 RQ |
|---|---|---|---|
| Topic preference | 愿意 / 看情况 / 不想聊哪些话题 | B 段、T01–T12 | RQ1 |
| Relationship-role fit | 某话题下希望哪个角色来陪（搭得上） | C 段、T×R 矩阵 | RQ1 |
| Role rejection | 哪个角色让人别扭、觉得假、不需要 | C 段、R1–R8（尤其 R8） | RQ1 |
| Cue preference | A / B / C 三种引入方式更接受哪种 | D 段、C1–C4 | RQ2 |
| Speaking pressure | 开口有没有压力、会不会怕被追问 | A/C/D 段 | RQ2 |
| Social presence | 有没有「有人陪着」的感觉 | C/D 段 | RQ2 |
| Cognitive load | 听不听得懂、跟不跟得上、会不会累 | 全程，尤其 D 段多角色 | RQ2（+可用性） |
| Memory boundary | 哪些可记 / 需确认 / 不可记 | E 段、M1–M5 | RQ3 |
| Ethical boundary | 敏感、隐私、故人、健康、依赖的红线 | 全程，尤其 T08/T09/T10 | RQ3 |
| Control need | 想不想自己主导、随时停、换角色、退场 | 全程，尤其 R8、喊停时 | RQ1 + RQ3 |

> **术语纪律：** 本编码本是 **researcher-only**。`multi-agent` / `relationship orchestration` / `agent-agent cueing` / RQ 编号等术语**只出现在研究者笔记里，绝不对参与者说**。

## 1. Topic preference（话题偏好）

**定义（抓什么）**
参与者对某个回忆话题「愿意讲 / 看情况讲 / 不想讲」的态度，以及**为什么**（触发点、情绪、禁忌）。这是最基础的一层，是话题—角色搭配（`Relationship-role fit`）的前提。

**示例编码 / 子码**
- 按态度三档：`[愿意]` / `[看情况]` / `[不想]`，后接 T-code。例：`[愿意:T05]`、`[不想:T09]`、`[看情况:T03]`。
- 按 T-code 记具体话题：`T01`…`T12`（沿用话题卡编号，不要另造名称）。
- 可选原因子码：`[破冰好开场]`（多见于 T04/T05/T06）、`[有话说/共同语言]`（T06/T11）、`[太私密]`、`[怕勾起情绪]`（T08/T09）、`[没什么好说]`。

**何时打这个码**
- 主要在 **B 段话题卡排序**：参与者把卡片分进三堆时，每一次分堆 + 每一句「为什么」都编一次。
- 也可在任意时刻：参与者主动说「这个我爱聊」「这个别聊了」时补打。
- **敏感话题只记态度、不追问深入**（T08/T09/T10 及 T03 家庭矛盾）；即使被放进「不想聊」，也只记编号与原话，不展开。

**关联**
- → `topic-cards.md`：T-code 与敏感度分档（低 / 中 / 高）以此为准。
- → `Relationship-role fit`：话题偏好确定后，才在「愿意/看情况」的话题上做话题—角色搭配。
- → `Ethical boundary`：凡涉及 T08/T09/T10/家庭矛盾的话题偏好，同时检查是否需并打 `Ethical boundary`。

## 2. Relationship-role fit（关系角色适配）

**定义（抓什么）**
在**某个具体话题**下，参与者希望由哪种关系角色（R1–R7）来陪，以及这种搭配为什么「合得来」。核心产出是**话题—关系角色矩阵**（记录表 §4）：一格格填「这个话题，老人愿意让哪个角色陪」。

**示例编码 / 子码**
- 用 `T→R` 记搭配：`[T05→R1]`（老照片配同龄共鸣者）、`[T02→R2/R3]`（工作经历配晚辈好奇者或中年传承者）、`[T08→R4]`（遗憾配长辈引导者）、`[T09/T10→R7]`（故人/健康配边界守护者）、`[收尾→R6]`（对话收尾配回忆整理者）。
- 强度子码：`[首选]` / `[可接受]` / `[勉强]`，用于同一话题多个角色时排序。
- 「换角色」信号：`[话题变→想换角色]`，记参与者是否希望话题一变就换陪伴角色。
- 角色速查（沿用角色卡）：`R1 同龄共鸣者` · `R2 晚辈好奇者` · `R3 中年传承者` · `R4 长辈引导者` · `R5 主题陪伴者` · `R6 回忆整理者` · `R7 边界守护者`（R8 属下面的 `Role rejection`/`Control need`）。

**何时打这个码**
- 主要在 **C 段话题—角色匹配**：每问一个「聊这个话题，您希望对方更像哪一种？」都编一次 `T→R`。
- 参与者对某角色明显有共鸣、点头认可、说「就该这样」时，打 `[首选]`。
- **注意区分适配 vs. 拒绝**：说「合得来」进本类；说「别扭 / 假 / 不需要」进 `Role rejection`。

**关联**
- → `role-cards.md`「话题—关系角色矩阵」规则 + 每张角色卡的「适用话题」。
- → `interview-record-template.md` §4（T×R 矩阵）：本类编码直接回填这张矩阵。
- → `Role rejection`：同一话题里被拒的角色记到下一类，两者互补。
- → `memory-card-examples.md` 对应表：R5→M2、R4/R6→M3、R7→M4/M5，可交叉印证。

## 3. Role rejection（角色拒绝）

**定义（抓什么）**
哪些角色让参与者觉得**别扭、假、被冒犯、或干脆不需要**，以及原因。这是 `Relationship-role fit` 的反面，同样重要——它划出「AI 角色不该越界的地方」，并检验 AI 角色是否被「默认接受」。**R8「不需要 AI 角色」是核心对照选项，选 R8 不是失败，是有价值的发现。**

**示例编码 / 子码**
- 按被拒角色：`[拒:R2]`、`[拒:R5]` 等，后接原因。
- 常见原因子码：`[像被盘问]`（多见 R2 连续追问）、`[显得假/尴尬]`、`[AI 抢话/秀知识]`（R1/R5 风险）、`[不合时宜]`（低敏感话题用了 R7 处处设防；或高敏感话题用了 R2 追问）。
- **R8 专门子码**（务必细分是哪一种「不需要」）：
  - `[R8:整体不要 AI]` —— 对 AI 陪伴整体有顾虑 / 更信任真人。
  - `[R8:这个话题不要 AI]` —— 仅某话题（常见 T08/T09/T10）不愿交给系统。
  - `[R8:这一刻不想要]` —— 当下情绪 / 状态原因，非稳定偏好。
  - `[R8:想自己讲]` —— 希望完全主导叙事，不要角色介入。

**何时打这个码**
- 主要在 **C 段**：参与者对某角色皱眉、摇头、说「这个不行 / 别扭 / 用不着」时。
- 出现 R8 表态时**必须编码**，且不要引导参与者改选别的角色，只如实记原因与是哪一种 R8。
- 也可在 D 段：多角色短对话（C 组的 C 方式）让人觉得「假、吵、插不上话」时——这既是 `Role rejection`（对多角色形式反感）也常并 `Cognitive load`。

**关联**
- → `role-cards.md` R8 说明（对照选项，不劝改选）+ 各角色「不适合场景」「边界提醒」。
- → `Control need`：R8「想自己讲 / 真人优先」与主导权、退场需求高度相关，常并打。
- → `Ethical boundary`：高敏感话题上选 R8，往往同时是一条边界信号。

## 4. Cue preference（引入方式偏好）

**定义（抓什么）**
对三种「把话题聊起来」的引入方式，参与者更接受哪一种、原因是什么。三种方式：**A 直接提问**（AI 直接问）/ **B 单角色铺垫**（一个 AI 先说一小段再邀请）/ **C 多角色短对话**（两三个 AI 先聊两句再请老人加入）。这是 RQ2 的核心测量：「多角色先短对话再邀请」是否比直接提问更好。

**示例编码 / 子码**
- 偏好排序：`[A>B>C]`、`[C>B>A]`、`[B>A=C]` 等，用 A/B/C 表示。
- 单项态度：`[C:自然]` / `[C:显吵/显假]` / `[A:干巴巴]` / `[B:刚好]`。
- 记来源脚本组：C1 老电视/旧物 · C2 老照片 · C3 工作/学习 · C4 新技术，例：`[C1: C>A]`。
- 可选：附上对比小结的印象打分（1–5，来自脚本文件末尾的可选评分）。

**何时打这个码**
- 主要在 **D 段引入方式对比**：每念完一组 C 脚本的 A/B/C 三条，参与者给的偏好与感受都编一次。
- 尽量让参与者**明确排序**（哪种最想接），排序本身就是编码内容。
- **区分维度**：本类只管「更喜欢哪种引入方式」；由此带出的「压力大不大」进 `Speaking pressure`，「有没有人陪着的感觉」进 `Social presence`，「几个 AI 一起说会不会乱」进 `Cognitive load`。

**关联**
- → `conversation-cue-scripts.md`：脚本组 C1–C4 与 A/B/C 三方式的定义、每组的「对比 A/B/C」提问。
- → `Speaking pressure` / `Social presence` / `Cognitive load`：三者是解释「为什么偏好某种引入方式」的三个常见理由，通常与本类成组出现。

## 5. Speaking pressure（开口压力）

**定义（抓什么）**
参与者开口讲述时**感到的压力或轻松程度**：会不会怕被追问、怕说错、怕冷场、觉得「非说不可」。低压力是本产品追求的体验之一（尤其 RQ2 想验证「多角色先聊、再邀请」能否降低开口门槛）。

**示例编码 / 子码**
- 方向：`[压力高]` / `[压力低/放松]` / `[中性]`。
- 来源子码：`[怕被追问]`（常见 R2 或 A 直接提问）、`[怕说错/怕冷场]`、`[被点名有压力]`、`[有人先起头就轻松]`（常见 B/C）、`[插一句就行]`（多见 C 多角色）。

**何时打这个码**
- **A 段**：问「一次聊多久舒服？会不会怕被追问？」时。
- **C 段**：某角色（如 R2 连续追问）让参与者感到压力时。
- **D 段**：对比 A/B/C 时，参与者说「这种有压力 / 这种没负担」——这是把引入方式偏好落到机制解释的关键。
- 一般作为 `Cue preference`、`Relationship-role fit`、`Role rejection` 的**伴随码**出现。

**关联**
- → `conversation-cue-scripts.md`（A/B/C 设计初衷就是比较开口压力）。
- → `Social presence`：低压力常与「有人陪着」的社交在场感一同出现，但两者是不同维度，需分开打码。
- → `study1-interview-guide-v2.md` §5：避免让参与者「为难」的访谈姿态，是压力信号的重要观察背景。

## 6. Social presence（社交在场感）

**定义（抓什么）**
参与者是否感到「有人陪着、有人在听、有回应」的**在场与陪伴感**，而非在对着一台冷冰冰的机器说话。这是 RQ2 里「多角色引入能否增强陪伴感」的正向测量。

**示例编码 / 子码**
- 方向：`[在场感强/像有人陪]` / `[在场感弱/像对机器]` / `[中性]`。
- 来源子码：`[几个人一起显热闹]`（多见 C 多角色的正面）、`[有人接话/被回应]`、`[像真的在聊天]`、`[太多声音反而假]`（C 的负面，此时并 `Role rejection`/`Cognitive load`）。

**何时打这个码**
- 主要在 **C 段 / D 段**：参与者对角色或多角色短对话表达「热闹 / 像有人陪 / 有回应」或反面感受时。
- 与 `Speaking pressure` 常同时出现，但**要分开**：「在场感」是「有没有人陪」；「压力」是「敢不敢开口」。可以在场感强但仍有压力，也可以轻松但觉得空。

**关联**
- → `conversation-cue-scripts.md`（C 方式多角色短对话正是为了营造在场感）。
- → `Well-being 边界`：在场感是好体验，但不得滑向「只有我懂您」的依赖式陪伴；一旦出现依赖信号，转 `Ethical boundary`。

## 7. Cognitive load（认知负荷）

**定义（抓什么）**
参与者在理解和跟上对话时的**认知负担**：话听不听得懂、几个角色一起说会不会乱、节奏会不会太快、访谈/对话会不会太长太累。这既服务 RQ2（多角色形式的可用性代价），也是重要的**可用性**信号。

**示例编码 / 子码**
- 方向：`[负荷高/跟不上]` / `[负荷正常]`。
- 来源子码：`[听不懂某问法]`（记下具体哪句话听不懂）、`[多角色分不清谁在说]`、`[节奏太快/插不上话]`、`[信息太多记不住]`、`[太长了/累了]`。

**何时打这个码**
- **全程**留意，尤其 **D 段多角色短对话**（C 方式）：几个 AI 一起说，最容易出认知负荷问题。
- 参与者反问「你说啥？」「刚才谁说的？」、明显走神、疲惫、要求重复时。
- 「哪些问题听不懂」是 pilot 记录表的必填项（见 `interview-record-template.md` 相关字段 / pilot 记录 8 字段），本类是它的编码归属。

**关联**
- → `Cue preference`：认知负荷是「不喜欢某种引入方式」的常见原因，尤其压低对 C 多角色的偏好。
- → `study1-interview-guide-v2.md` §3（不要说术语）：很多「听不懂」其实是术语泄漏或问法太绕，编码时要区分「话题本身难」还是「问法有问题」，后者直接反馈到材料修订。

## 8. Memory boundary（记忆边界）

**定义（抓什么）**
参与者对「哪些内容可以被系统长期记住」的态度与边界：可记 / 需先确认 / 改了再记 / 坚决不记 / 以后别再提。对应记忆卡片 M1–M5 与四个固定选项。这是 RQ3 长期记忆设计的第一手依据。

**示例编码 / 子码**
- 四选一（沿用卡片固定选项）：`[保存]` / `[修改后保存]` / `[不保存]` / `[以后不要再提]`，前缀记忆类型，如 `[M4:不保存]`、`[M2:修改后保存]`。
- 记忆类型（沿用记忆卡片）：`M1 事实记忆`（保存前确认）· `M2 兴趣记忆`（可保存/可改）· `M3 情感记忆`（谨慎保存）· `M4 敏感记忆`（默认不保存/强确认）· `M5 边界偏好记忆`（可保存，用于保护老人）。
- 过程子码：`[主动改措辞]`（去细节 / 去病名 / 淡化）、`[犹豫]`、`[要求只记一句]`、`[频率偏好]`（如 M3「别老提」）、`[愿意共同校对]`（支持 R6 确认流程）。

**何时打这个码**
- 主要在 **E 段记忆与边界**：每张记忆卡片（M1–M5）的四选一结果 + 参与者怎么改措辞，都编一次。
- 参与者说「这个可以记」「这个别记」「记的时候把 X 去掉」时。
- **M4 及涉及 T08/T09/T10/家庭矛盾的记忆默认落 `[不保存]`**，需明确强确认才记；此类几乎总要并打 `Ethical boundary`。

**关联**
- → `memory-card-examples.md`：M1–M5 类型、四个固定选项、默认处理（越敏感越保守）、以及记忆类型 × T-code × R-code 对应表。
- → `Ethical boundary`：M4 敏感记忆与 M5 边界偏好几乎总与伦理红线并存。
- → `Control need`：`[以后不要再提]` 既是记忆边界，也是「参与者主导、系统退让」的控制信号。

## 9. Ethical boundary（伦理边界）

**定义（抓什么）**
涉及**敏感、隐私、故人、健康、依赖**的红线信号：参与者在哪里踩刹车、要求停下、不愿被追问，以及系统必须遵守的安全边界是否被参与者认可。这是 RQ3 里「系统如何解释、确认、暂停、婉拒」的关键证据，也是安全底线的守门码。

**示例编码 / 子码**
- 触发场景：`[敏感话题]`（T08/T09/T10/家庭矛盾）、`[隐私]`、`[哀伤/故人]`（T09）、`[健康担忧]`（T10）、`[依赖信号]`（如「只有你懂我」「别走」）。
- 系统边界确认：`[认可:绝不扮演逝者]`、`[认可:不诊断不给用药]`、`[认可:随时可停]`、`[期望:被保护/踩刹车]`（对应 R7）。
- 违规/风险观察（供材料自查）：`[风险:问法像追问细节]`、`[风险:共鸣加深孤独]`、`[风险:术语/医疗越界]`——用于回头修材料，不针对参与者。

**何时打这个码**
- **全程**，尤其一切触及 T08/T09/T10 与 T03 家庭矛盾的时刻。
- 参与者喊停、跳过、沉默、情绪波动、要求「别问了」时。
- **安全红线必须无条件遵守并编码**：T09 绝不扮演逝者、不追问离世细节；T10 不诊断、不给用药剂量建议；出现依赖信号时温和引向真人支持（well-being over stickiness）。
- 记录参与者对这些边界的**认可度**（是否觉得「这样很好 / 该这样」），是设计安全护栏是否被接受的直接证据。

**关联**
- → `study1-interview-guide-v2.md` §5 敏感话题处理原则（不强行深入、绝不扮演逝者、不诊断、避免加深孤独）。
- → `memory-card-examples.md` M4/M5 的安全与边界条款、`role-cards.md` R7 边界守护者（及各角色通用安全红线）。
- → `Memory boundary` / `Control need` / `Role rejection`：伦理边界常与这三者叠加出现，编码时一并检查。

## 10. Control need（控制/主导需求）

**定义（抓什么）**
参与者希望**自己掌握对话主导权**的程度：想不想自己讲、随时喊停、换个角色、决定聊多深、乃至让 AI 完全退场（R8）。它横跨 RQ1（要不要 AI 角色、要哪个）与 RQ3（暂停 / 婉拒 / 退让的边界）。

**示例编码 / 子码**
- 主导偏好：`[想自己主导讲]`、`[想被引导着讲]`、`[看情况]`。
- 控制动作需求：`[要能随时停]`、`[要能换角色]`、`[要能删掉/别再提]`（连 `[以后不要再提]`）、`[要真人而非 AI]`（连 R8）。
- 退场信号：`[这个话题要 AI 退场]`、`[整体想少一点 AI 介入]`。

**何时打这个码**
- **全程**，尤其：A 段「更喜欢被问还是自己讲」；C 段出现 R8 或「话题变想换角色」；E 段选 `[以后不要再提]`；任何一次喊停、跳过、要求换话题。
- 参与者强调「我说了算」「我想停就停」「我想自己讲」时。

**关联**
- → `role-cards.md` R8（不需要 AI 角色，对照选项，不劝改选）+ R7（据边界偏好暂停 / 转向 / 婉拒）。
- → `Role rejection` [R8:*] 与 `Memory boundary` [以后不要再提]：常与本类同时成立。
- → `study1-interview-guide-v2.md`「由参与者主导」原则 + well-being over stickiness（真人优先时把老人引向家人 / 朋友 / 专业支持）。

## 11. 多码并存与优先级（快速判断）

一段话经常同时命中多个码，这是正常的，应**全部标注**。几条实用判断：

- **凡触及 T08/T09/T10/家庭矛盾** → 除相应内容码外，**必并 `Ethical boundary`**，并检查是否要并 `Memory boundary`（多为 M4）。
- **选了 R8 或要求 AI 退场** → 并 `Role rejection [R8:*]` + `Control need`；若发生在敏感话题，再并 `Ethical boundary`。
- **对某引入方式（A/B/C）表态** → 主码 `Cue preference`；理由分别落到 `Speaking pressure` / `Social presence` / `Cognitive load`。
- **喊停 / 跳过 / 沉默** → 至少 `Ethical boundary` + `Control need`；若和记忆卡片有关再并 `Memory boundary`。
- **「听不懂 / 问法太绕」** → `Cognitive load`，并回头判断是「话题难」还是「材料/术语问题」（后者反馈到材料修订）。

## 12. 编码到分析的去向（pilot 阶段）

- `Topic preference` / `Relationship-role fit` / `Role rejection` → 汇总成**话题—关系角色矩阵**（记录表 §4），是 RQ1 的核心产出。
- `Cue preference` / `Speaking pressure` / `Social presence` / `Cognitive load` → 汇总成 **A/B/C 引入方式对比**，回答 RQ2。
- `Memory boundary` / `Ethical boundary` / `Control need` → 汇总成**记忆与边界规则**，回答 RQ3，并直接约束 R7 边界守护者与 M4/M5 的默认处理。

> pilot 数据量很小（1–2 人），**只做定性归纳、不做统计推断、不下「哪种更好」的结论**。目标是验证材料与编码框架是否可用。

## 13. pilot 后如何修订本编码本

本编码本是**初步版本**。做完 pilot，用 pilot 记录 8 字段（哪些问题听不懂 · 哪些话题愿意聊 · 哪些话题明显敏感 · 哪些角色自然 · 哪些角色尴尬 · 多角色短对话是否可接受 · 访谈是否太长 · 需要删除或改写的问题）回看这 10 类编码，检查：

1. **有没有反复出现、却无处安放的内容？** → 可能要新增类别或子码。
2. **某个子码是否从没被用到？** → 考虑删除或合并。
3. **两类编码是否总是纠缠不清、难以区分？** → 补写「区分维度」说明，或合并。
4. **哪些「听不懂 / 尴尬」其实是材料问题？** → 反馈到 `topic-cards.md` / `role-cards.md` / `conversation-cue-scripts.md` 的措辞修订，而非只改编码本。
5. 修订后升版本号，并在此处记录改了什么、为什么。

> **一句话记住：** 编码本服务于「听懂老人怎么想」，而不是把老人硬塞进既有格子。格子不够用，就改格子。
