## 0. 阅读前提：角色 vs 智能体（不要混淆）

“角色（Role, R1–R7）”是**关系视角（relationship perspective）**，实现在**陪伴层（CompanionAgent 的表现层）**上，**不是新的自治智能体**。

代码里仍然只有四个自治智能体：`CoordinatorAgent` · `CompanionAgent` · `GuardianAgent` · `SafetyCriticAgent`（见 `CLAUDE.md` §8）。C1/C2/C3 的差异**不在**于新增智能体，而在于：

- 同一层（陪伴层）上**呈现几个关系视角气泡**；
- 这些视角的**集合是固定的还是按话题/边界动态选出来的**；
- 视角之间**是否先互相对话（agent-agent cueing）再邀请老人加入**；
- 编排来源是**固定脚本**还是 **WoZ 研究者后台 / 半自动系统**。

> 术语约定：下文说“AI 角色气泡”指的就是这些呈现在陪伴层上的关系视角，不代表底层多进程智能体。

### 关系角色清单（来自 Study 1，跨引用 `docs/research/study1/role-cards*`）

| 代号 | 名称 | 关系视角简述 |
|---|---|---|
| R1 | 同龄共鸣者 | 用“我们那一代”的共同经历共情、附和、接话 |
| R2 | 晚辈好奇者 | 以晚辈口吻好奇发问，请老人多讲一点 |
| R3 | 中年传承者 | 关注“这段经历能留给后人什么”，承接与传递 |
| R4 | 长辈引导者 | 以更年长者口吻温和引导、给节奏，不评判 |
| R5 | 主题陪伴者 | 围绕当前主题稳定陪聊，负责话题连续性 |
| R6 | 回忆整理者 | 把零散回忆归纳成可确认的“记忆卡片” |
| R7 | 边界守护者 | 处理故人 / 健康 / 隐私 / 依赖信号；**绝不扮演逝者** |
| R8 | 不需要 AI 角色 | 老人此刻不想要任何 AI 角色介入（可作为选择/退出信号） |

## 1. C1 — 单智能体直接提问（Single-agent）

### 定义
一个陪伴 agent **直接**面向老人提问，无角色分化、无 agent-agent 对话。这是最小对照基线：只有“材料 → 一个声音的提问 → 老人回答”。

### 对话形式（每一轮发生什么）
1. 呈现话题材料（照片 / 实物 / 歌曲 / 话题卡，见 §5 恒定变量）。
2. 系统以**单一陪伴声音**发出一个开放式问题（例如“这张照片里是在哪儿呀？”）。
3. 老人回答。
4. 同一个声音基于回答**继续追问**，循环推进。
- 无角色铺垫、无角色署名、无角色间对白。

### 角色数量与角色集合
- **气泡数量：恒为 1。**
- 角色集合：`{单一陪伴视角}`（在数据里可记为 `selected_roles = ["C1_single"]` 或空占位；不映射到 R1–R7 的具体分化）。

### 编排来源
- **固定脚本 / 固定提问策略。** 无编排器动态选人。WoZ 阶段研究者仅按脚本推进追问，不做角色选择决策。

### 老人看到什么
- 屏幕上**始终只有 1 个气泡**在提问。
- **没有**任何角色之间的对话；不出现“换一个人聊”产生的第二种声音。

### 保持恒定的变量
- 材料集合、单次时长、话题顺序、边界处理（R7 逻辑仍在后台生效，见 §6）与 C2/C3 一致。

### 验证目的（映射 RQ）
- 作为 **基线**，用来分离“多智能体 / 角色分化本身”带来的增量。
- 对照 RQ2（多角色是否比单一提问更让老人愿意讲、感到被理解）与 RQ3（关系感知是否比无差别提问更少打扰、更被接纳）的**下限锚点**。

## 2. C2 — 固定角色多智能体（Fixed-role multi-agent）

### 定义
用**固定的两个关系角色**做铺垫：**R1 同龄共鸣者 + R2 晚辈好奇者**。角色集合**不随话题动态变化**——每一轮、每个话题都是这同两个角色。

### 对话形式（每一轮发生什么）
1. 呈现话题材料。
2. **R1 同龄共鸣者**先以同龄口吻共情/接话（例如“这我们那时候也常这样”）。
3. **R2 晚辈好奇者**再以晚辈口吻好奇发问（例如“那后来呢，您能多讲讲吗？”）。
4. 老人回答。
5. 两个固定角色按**固定脚本**继续（共鸣 → 好奇提问）循环。
- 可以有**极短的角色间铺垫**（R1 说完 R2 接），但角色集合和顺序是**写死的**，不是按话题选出来的。

### 角色数量与角色集合
- **气泡数量：恒为 2。**
- 角色集合：**固定** `selected_roles = ["R1", "R2"]`，所有话题一致。

### 编排来源
- **固定脚本。** 无动态选择、无关系偏好决策。WoZ 阶段研究者仍只按固定的 R1→R2 顺序推进，不替换角色。

### 老人看到什么
- 屏幕上**恒为 2 个气泡**（R1、R2）。
- 角色之间**有一点点铺垫式衔接**（同龄共鸣 + 晚辈好奇的固定二人组），但**没有按话题变化的角色阵容**，也没有“先短对话再邀请老人加入”的动态 cueing。

### 保持恒定的变量
- 与 C1/C3 相同的材料、时长、话题顺序、后台边界处理（R7）。

### 验证目的（映射 RQ）
- 隔离“**多角色分化本身**”的效果（vs C1 的单声音）——对应 RQ2 的中间锚点。
- 作为 C3 的对照：C2 提供“多角色但**不感知关系/话题**”，因此 C2↔C3 的差异**纯粹来自动态角色选择 + agent-agent cueing**，直接服务 RQ3。

## 3. C3 — 关系感知多智能体（Relationship-aware multi-agent）

### 定义
编排器根据**话题、记忆、关系偏好和边界风险**，**动态**从 R1–R7 中选出 **2–3 个角色**，并支持 **agent-agent cueing**（角色之间先进行简短对话，再邀请老人加入）。编排来源在 WoZ 阶段是**研究者后台**，在 semi-auto / auto 阶段才逐步交给系统。

### 对话形式（每一轮发生什么）
1. 呈现话题材料。
2. 编排器（WoZ=研究者后台 / semi-auto=系统建议+研究者确认）读取**话题、可用记忆、老人的关系偏好、边界风险信号**，**动态选出 2–3 个角色**（例如“故人”话题 → R5 主题陪伴 + R7 边界守护，且 R7 绝不扮演逝者）。
3. **被选中的角色之间先进行简短对话（agent-agent cueing）**——例如 R1 对 R2 说“你要不要问问她当年那件事”，R2 应“好呀，我正想听”。
4. **然后邀请老人加入**这段对话（“奶奶，您愿意跟我们说说吗？”）。
5. 老人回答后，编排器可**换人、增减角色、或触发 R6 回忆整理者**产出记忆卡片，并**始终**让 R7 边界守护者在后台监听风险。

### 角色数量与角色集合
- **气泡数量：2–3 个，逐轮可变。**
- 角色集合：从 **R1–R7 动态子集**中选取；集合与理由都要落到 trace（`selected_roles` + `role_selection_reason`，见 §7）。
- **R7 边界守护者可在任何条件下被拉入**，但在 C3 下它是**动态编排的一部分**（可见气泡或后台干预）。

### 编排来源
- **WoZ 阶段：研究者后台**手动选择角色并触发 cueing（对老人透明地表现为“系统在协调”）。
- **semi-auto 阶段：系统给出角色建议 + cueing 草稿，研究者一键确认/微调。**
- **auto 阶段（后续，未实现）：系统自动决策。** 本文件不声称 auto 已完成。

### 老人看到什么
- 屏幕上出现 **2–3 个气泡**，**阵容随话题/边界变化**。
- **能看到角色之间的短对话**（agent-agent cueing）——这是 C3 相对 C1/C2 最直观的可见差异——**然后**被邀请加入。
- 触发边界（故人/健康/隐私/依赖）时，R7 以温和方式介入，**不扮演逝者、不给诊断/剂量、不触发真实紧急**。

### 保持恒定的变量
- 与 C1/C2 相同的材料集合、单次时长、话题顺序。**变化的只有**角色选择策略和是否有 agent-agent cueing。

### 验证目的（映射 RQ）
- 直接检验主 RQ 的核心命题：**关系感知的动态角色编排 + 可解释协作**是否比无差别提问（C1）/固定角色（C2）**更提升愿意讲述、被理解感、信任，同时不增加被打扰感与依赖风险**。
- C1→C2 分离“多角色”，C2→C3 分离“**关系感知 + agent-agent cueing**”，两级对照共同支撑 RQ1（可解释/控制感）、RQ2（陪伴/被理解）、RQ3（主动关怀适恰性 / 不过度打扰）。

## 4. 并排对比表（维度 × C1/C2/C3）

| 维度 | C1 单智能体直接提问 | C2 固定角色多智能体 | C3 关系感知多智能体 |
|---|---|---|---|
| 一句话定义 | 一个声音直接问 | 固定 R1+R2 铺垫 | 按话题/边界动态选 2–3 角色 + cueing |
| 气泡数量 | 恒 1 | 恒 2 | 2–3，逐轮可变 |
| 角色集合 | 单一视角（不分化） | 固定 `{R1, R2}` | 动态子集 ⊂ `{R1…R7}` |
| 角色是否随话题变 | 否 | 否 | **是** |
| agent-agent cueing（先角色互聊再邀请老人） | 无 | 无（仅固定二人衔接） | **有** |
| 编排来源 | 固定脚本 | 固定脚本 | WoZ 研究者后台 → semi-auto → (auto 未实现) |
| 关系偏好 / 记忆参与选角 | 否 | 否 | **是**（记忆 + 关系偏好 + 边界风险驱动选角） |
| 老人看到角色间对话 | 否 | 否 | **是** |
| R7 边界守护（故人/健康/隐私/依赖） | 后台生效 | 后台生效 | 后台 + 可作为动态选中的可见角色 |
| 绝不扮演逝者 | 是 | 是 | 是 |
| 恒定变量（材料/时长/话题顺序） | 与 C2/C3 一致 | 与 C1/C3 一致 | 与 C1/C2 一致 |
| 主要对照作用 | 基线下限 | 分离“多角色本身” | 分离“关系感知 + cueing” |
| 复用的自治智能体 | Companion(+后台 Safety) | Companion(+后台 Safety) | Coordinator 编排缝 + Companion + Guardian + SafetyCritic |

> **C3 之所以不同，只有两点：动态角色选择（selection）+ agent-agent cueing。** 其余一切（材料、时长、话题、边界纪律）都保持恒定，以便把差异干净地归因给这两点。

## 5. 三条件恒定变量（Constants across C1/C2/C3）

为保证内部效度，以下变量在三条件间**保持一致**（跨引用 study2 材料与协议文档）：

- **材料**：同一套话题材料与呈现方式（照片 / 实物 / 歌曲 / 话题卡 T01–T12，见 `docs/research/study1/topic-cards*` 与 `topic-role-matrix`）。
- **单次时长 / 话题数量**：每条件相同（具体值在 study2 协议文档定，本文件不写死）。
- **话题顺序**：按平衡设计安排，跨条件一致或对称抵消。
- **边界纪律**：R7 逻辑与安全模板在三条件后台**全程一致生效**（见 §6）。
- **操作者脚本纪律**：WoZ 操作者在三条件下遵循各自的固定脚本 / 决策手册，避免操作者变异污染条件差异。

## 6. 边界处理在三条件都适用（R7，绝不扮演逝者）

边界处理**不是** C3 专属；**C1、C2、C3 全程一致适用**（复用现有安全栈：`InputRuleGuard`/`OutputRuleGuard` 常开 + 触发风险时 `SafetyCriticAgent` + 安全模板 + mock 紧急联系人）。

R7 边界守护者处理 **故人 / 健康 / 隐私 / 依赖** 信号，必须：

- **绝不扮演逝者**（不以逝者口吻说话、不代逝者“回话”）；
- **不做诊断、不给药物剂量、不触发真实紧急**（demo 明确说明不拨打真实电话）；
- 以温和、克制、非评判的方式回应，可提供安全的替代帮助（记提醒、展示 mock 紧急联系人、建议联系家人/医生/药师）。

差异**仅在于呈现**：C1/C2 中 R7 在后台生效（不额外增加可见气泡数量）；C3 中 R7 可作为**动态选中的可见角色**参与。触发的边界信号在三条件都要写入 `boundary_flags`（见 §7）。

## 7. 与 trace 字段的对应（本文件只给最小对应，细化见 study2 trace 需求）

条件规格通过后台 trace 记录以支撑分析。各字段的完整类型/取值在 study2 trace 需求文档定义；此处给出与条件相关的最小对应：

- `condition`：`C1 | C2 | C3` —— 标注本文件定义的条件。
- `selected_roles`：C1 = 单一占位；C2 = 固定 `["R1","R2"]`；C3 = 动态子集 ⊂ `{R1…R7}`。
- `role_selection_reason`：仅 C3 有实质内容（话题/记忆/关系偏好/边界风险如何驱动选角）；C1/C2 记为“固定脚本”。
- `cueing_style`：C1 = `direct_question`；C2 = `fixed_prelude`；C3 = `agent_agent_cueing`（命中边界转 `boundary_redirect`；`none` 作兜底）。
- `boundary_flags`：三条件都记录（故人/健康/隐私/依赖 触发情况）。
- `memory_used` / `memory_card_action`：C3 中记忆参与选角与卡片确认时记录；C1/C2 按实际发生记录。`memory_card_action` 取值 `none | save | edit_then_save | do_not_save | never_mention_again`（对应 Study 1 记忆卡片四选项）。
- `user_control_actions`：三条件都记录老人使用控制按钮（继续 / 换一个人聊 / 先别问了 / 帮我总结 / 不要记住）的情况。

> 说明：控制按钮“换一个人聊”在 C1 语义上应被禁用或降级（C1 恒 1 个声音）；在 C2 表示在固定二人组内切换；在 C3 表示请编排器重选角色。此差异也需落到 `user_control_actions` 以便区分条件。

## 9. 范围与安全声明（Scope & Safety）
- 本文件为**需求/规格**，不含代码实现；不涉及参与者招募。
- **不声称 C3 自动编排已完整实现**——WoZ / semi-auto 先行。
