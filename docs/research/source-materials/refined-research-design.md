# 关系感知型多智能体老年陪伴系统：细化版研究方案

## 0. 一页概览

### 暂定题目

关系感知型多智能体老年陪伴系统：面向回忆、自我表达与长期信任的动态角色编排

英文暂定：

Relationship-Aware Multi-Agent Companions for Older Adults' Reminiscence, Self-Disclosure, and Long-Term Trust

### 一句话版本

本研究关注老年回忆陪伴场景中，系统如何根据老人的话题、人生经历、关系偏好和敏感边界，动态编排不同 AI 陪伴角色，并验证多个 AI 之间的短暂对话能否作为一种社会性记忆线索，更自然地引导老人进入回忆和自我表达。

### 不是做什么

本研究不是简单做一个“陪老人聊天的 AI”，也不是把多个 agent 同时放进聊天框。

也不是马上开始完整开发一个大系统。当前阶段更重要的是把研究问题、访谈设计、文献依据、系统机制和后续实验条件细化清楚。

### 主要做什么

1. 通过访谈和共创理解老年人在不同话题下希望由什么样的“关系角色”陪伴。
2. 设计一种关系感知的多智能体编排机制：谁说话、谁沉默、谁追问、谁总结、何时停止。
3. 验证 agent-agent conversation 是否可以作为社会性记忆线索，降低老人开口压力，提供更多回忆入口。
4. 比较 single-agent、fixed-role multi-agent 和 relationship-aware multi-agent 三种方式对回忆、自我表达、陪伴感、信任感和控制感的影响。

## 1. 老师说“细化”后的理解

老师让细化，重点不是“现在立刻把完整系统做出来”，而是要把方向从概念变成可研究、可执行、可验证的方案。

细化后需要能回答：

1. 研究对象是谁？
2. 具体研究什么机制？
3. 前人已经做了什么？
4. 前人还没解决什么？
5. 我们的方法如何从前人工作推出？
6. 访谈要问什么、得到什么？
7. 后续原型和实验怎么设计？
8. 最后用什么指标说明这个系统更好？

因此当前最应该推进的是：

> 文献矩阵 + 访谈设计 + 角色机制定义 + 后续实验方案，而不是马上做完整产品。

## 2. 背景与问题收束

老年人回忆过去并不只是怀旧。回忆可能服务于自我认同、情绪调节、经验传承、社会连接和人生意义整理。已有研究说明照片、旧物、音乐、地点、虚拟场景和对话都可以作为记忆线索。

但是传统回忆活动通常依赖真实的人类陪伴者，例如家人、朋友、同龄人或照护者。这些陪伴并不总是稳定可得。同时，普通 AI 聊天系统往往以单一角色直接提问，容易出现三个问题：

1. **关系不匹配**：不同话题适合不同倾听者，但单一 agent 难以同时承担同龄共鸣、晚辈好奇、人生总结和边界守护等功能。
2. **话题进入生硬**：AI 直接问“你想起什么了吗”可能让老人有压力，也可能缺少自然的社会语境。
3. **多角色缺少编排**：已有多角色系统可以提供不同身份，但如果缺少发言顺序、沉默机制和总结机制，多个 agent 可能变成噪声。

所以本研究要把问题收束为：

> 老年回忆陪伴中，不同“关系角色”如何被动态编排，才能更自然地支持老人进入回忆、自我表达和长期信任？

## 3. 研究缺口

### 3.1 老年回忆技术重视“线索”，但较少研究“谁来引入线索”

已有系统常用照片、旧物、音乐、VR 场景等材料触发回忆。这说明外部线索很重要。但它们更多关注“什么东西能触发回忆”，较少研究：

- 由谁来提出这个话题更自然？
- 同龄人、晚辈、中年人、主题专家等不同关系角色会不会影响老人愿不愿意讲？
- 多个 AI 之间先短暂互动，是否能创造更自然的多人聊天氛围？

### 3.2 多 agent 研究重视协作能力，但老年陪伴场景需要低负担编排

多 agent 系统常用于创意、研究、写作、软件开发等任务，重点是专家分工、协作生成、辩论和任务完成。但老年陪伴不是让用户管理一个复杂团队。

老年用户可能更需要：

- 少量可理解的角色
- 简单的控制方式
- 自然的发言轮替
- 可以随时停止或换话题
- 不被多个 agent 同时打扰

因此，多 agent 的价值不是“更多 agent”，而是“更合适的角色在更合适的时机出现”。

### 3.3 ReminiBuddy 证明多角色有价值，但角色仍然固定

ReminiBuddy 使用 older / younger agents 和照片/3D 怀旧物件支持老年回忆，说明 agent 身份和记忆线索会影响回忆体验。

但它还有可继续推进的空间：

1. 角色主要是固定的 older / younger。
2. agents 之间没有真正形成对话。
3. 没有系统化研究“话题-关系角色”匹配。
4. 对 agent-agent conversation 作为记忆线索的效果还缺少验证。

本研究可以在它的基础上进一步问：

> 如果角色不是固定的，而是根据话题、个人经历、长期记忆和边界需求动态编排，会不会更适合老年人的回忆与自我表达？

## 4. 核心概念定义

### 4.1 关系角色

关系角色不是指真实家人，也不是让 AI 假装成某个具体的人，而是不同的陪伴功能和社会位置。

初步角色包括：

| 角色 | 主要功能 | 适合话题 |
|---|---|---|
| 同龄共鸣者 | 提供时代共鸣、共同经历感 | 旧物、工作、集体生活、老照片 |
| 晚辈好奇者 | 以好奇方式追问细节，降低表达压力 | 家庭故事、旧习俗、人生经验 |
| 中年传承者 | 承接经验、总结价值、理解责任 | 工作、家庭教育、人生选择 |
| 长辈引导者 | 温和支持人生反思 | 遗憾、重要决定、情绪性话题 |
| 主题陪伴者 | 提供内容共同语言 | 戏曲、地方文化、职业、老电影 |
| 回忆整理者 | 整理故事、生成记忆卡片 | 对话结束、故事总结 |
| 边界守护者 | 识别隐私、悲伤、依赖风险 | 故人、家庭矛盾、健康、强烈情绪 |

### 4.2 关系编排

关系编排指系统根据当前话题、老人偏好、历史记忆、情绪状态和边界风险，决定：

- 当前应该由谁说话
- 谁应该保持沉默
- 是否需要多个 AI 先短暂对话
- 是否应该追问、共鸣、解释或总结
- 是否应该暂停、转向或询问用户是否继续

关系编排不是把复杂机制展示给老人，而是在后台完成。前台只保留低负担控制：

- 换一个人聊
- 继续说
- 先别问了
- 帮我总结
- 不要记住
- 不要再提这个

### 4.3 社会线索引入话题

社会线索引入话题指：

> 不是由单个 AI 直接提问，而是让两个或三个 AI 先围绕照片、旧物或话题短暂互动，形成一个轻量的多人聊天场，再邀请老人加入。

示例：

> 同龄共鸣者：看到这个老电视，我会想到以前谁家有电视，邻居们可能都会过去看。
> 晚辈好奇者：真的会大家一起看吗？那是不是还要提前搬凳子？
> 中年传承者：这不只是看电视，也像是一种邻里关系。叔叔/阿姨，您那时候有没有类似的经历？

它要验证的问题是：

- 这种方式是否比直接提问更自然？
- 是否降低开口压力？
- 是否提供更多回忆入口？
- 是否会让老人觉得太吵、太假或被引导过多？

### 4.4 多智能体角色交互

多智能体角色交互不是所有 agent 同时发言，而是有节奏的协作：

| 阶段 | 角色分工 |
|---|---|
| 话题启动 | 同龄共鸣者或主题陪伴者提供熟悉线索 |
| 细节展开 | 晚辈好奇者追问具体人、地点、时间、物件 |
| 意义承接 | 中年传承者总结经验、价值和今昔变化 |
| 情绪敏感 | 长辈引导者放慢节奏，边界守护者判断是否继续 |
| 对话收束 | 回忆整理者生成记忆卡片，并询问是否保存 |

## 5. 研究问题

### 主研究问题

相比单一陪伴 agent 或固定多角色 agent，关系感知型多智能体编排是否能更好地支持老年人的回忆启动、自我表达、陪伴感、信任感和控制感？

### RQ1：关系编排

老年人在不同回忆话题下希望由哪些关系角色陪伴？系统能否根据话题、个人经历和关系偏好动态选择合适角色？

### RQ2：社会线索引入话题

多个 AI 先进行短暂对话，再邀请老人加入，是否比 AI 直接提问更能降低开口压力、提供更多记忆入口，并增强陪伴感？

### RQ3：多智能体角色交互

不同关系角色如何分工与协作，才能避免信息过载，同时支持共鸣、追问、总结和边界保护？

### RQ4：边界与控制

当话题涉及故人、悲伤、隐私或过度依赖风险时，系统应如何解释、暂停、转向或拒绝继续？

## 6. 文献如何推出我们的方法

| 文献/方向 | 已有做法 | 还没解决什么 | 我们怎么做 |
|---|---|---|---|
| reminiscence / life review | 说明回忆对身份、情绪、意义和社会连接有价值 | 没有回答 AI 应该以什么关系位置陪老人回忆 | 把目标定为回忆、自我表达、经验传承和陪伴 |
| 技术支持回忆 | 用照片、旧物、音乐、VR 等线索触发回忆 | 多数线索是物件或视觉材料，社会关系线索较少 | 把 agent-agent conversation 设计成社会性记忆线索 |
| Photo-based reminiscence | 用访谈和共创理解老人如何使用照片 | 主要关注照片，不关注多角色陪伴 | 访谈同时研究话题、角色、记忆和边界 |
| ReminiBuddy | older / younger agents + 照片/3D 物件 | 角色固定，agent 之间不真正对话 | 做动态关系编排和 agent-agent cueing |
| Perspectra / CrafTeam | 用户选择或编排多个 expert agents | 面向研究/创意用户，控制复杂 | 转化为老人可理解的低负担控制 |
| MultiColleagues / YES AND | 多 agent 对话和 turn-taking 促进协作 | 场景不是老年回忆 | 把 turn-taking 转成陪伴节奏 |
| AGDebugger / DiLLS | 多 agent 系统需要 trace 和诊断 | 面向开发者，不面向老人 | 后台记录 role/topic/memory/boundary trace，用于研究分析 |
| 信任与自动化文献 | 说明信任需要校准和控制 | 陪伴 AI 可能带来依赖或误信 | 设计边界守护、记忆确认和删除机制 |

由此推出的方法链：

> 文献综述 → 半结构访谈 → 话题卡与角色卡共创 → 关系角色需求分类 → 关系编排规则 → Wizard-of-Oz 或半自动原型 → 对比实验 → 内容编码与体验评估

## 7. 系统机制细化

### 7.1 输入信息

系统编排时可考虑以下输入：

| 输入 | 例子 | 作用 |
|---|---|---|
| 当前话题 | 老照片、工作、家庭、故人、新技术 | 判断适合的关系角色 |
| 记忆材料 | 照片、旧物、歌曲、地方文化图片 | 提供回忆入口 |
| 用户偏好 | 喜欢同龄人、不喜欢连续追问 | 调整角色和发言方式 |
| 长期记忆 | 过往职业、兴趣、重要地点 | 提高回应贴合度 |
| 情绪状态 | 开心、沉默、难过、抗拒 | 决定是否追问或放慢 |
| 边界风险 | 故人、隐私、健康、家庭冲突 | 触发边界守护 |

### 7.2 编排输出

系统每一轮需要决定：

1. 选择哪个主发言角色。
2. 是否加入辅助角色。
3. 是否使用 agent-agent cue。
4. 追问细节、共鸣、解释、总结还是暂停。
5. 是否生成记忆卡片。
6. 是否询问用户是否保存或继续。

### 7.3 发言规则

为了避免多个 AI 抢话，可以先设定简单规则：

| 情况 | 编排规则 |
|---|---|
| 话题刚启动 | 最多 2-3 个 AI 短暂互动，每个 1 句 |
| 老人开始讲述 | 其他 AI 保持沉默，由一个角色接话 |
| 老人讲得很具体 | 晚辈好奇者追问 1 个细节，不连续追问 |
| 老人讲到经验或价值 | 中年传承者总结，但要询问是否准确 |
| 老人沉默或回答短 | 换更低压力的问题，或提供可选入口 |
| 老人表达不适 | 立即停止追问，询问是否换话题 |
| 对话结束 | 回忆整理者总结，并请求保存许可 |

### 7.4 社会线索引入流程

1. 识别当前材料或话题。
2. 判断该话题是否适合 agent-agent cue。
3. 选择 2-3 个角色。
4. 生成 2-3 句短对话。
5. 明确邀请老人加入。
6. 根据老人反应决定继续追问、总结或换话题。

不适合 agent-agent cue 的情况：

- 话题已经很私密
- 老人明显难过
- 老人表示不想继续
- 话题涉及故人、家庭矛盾、健康隐私
- 用户已经开始主动讲述，不需要 AI 预热

### 7.5 记忆与边界机制

系统不能默认保存所有内容。建议分级：

| 记忆类型 | 例子 | 默认处理 |
|---|---|---|
| 兴趣记忆 | 喜欢戏曲、老电影、种花 | 可保存，但允许修改 |
| 事实记忆 | 曾经做过教师、住过某地 | 保存前简短确认 |
| 情感记忆 | 讲到某段经历很骄傲 | 谨慎保存，避免频繁提起 |
| 敏感记忆 | 故人、疾病、家庭矛盾 | 默认不保存或强确认 |
| 边界偏好 | 不想聊某话题、不喜欢某角色 | 可保存，用于保护用户 |

## 8. Study 1：访谈与共创

### 8.1 目的

Study 1 的目标不是证明系统有效，而是建立设计依据：

1. 老人回忆时真实需要什么样的陪伴关系。
2. 不同话题适合哪些 AI 角色。
3. 老人是否接受多个 AI 先聊两句再邀请自己加入。
4. 哪些记忆可以保存，哪些需要确认或禁止保存。
5. 哪些话题需要边界守护。

### 8.2 参与者

计划访谈 8-12 位 60 岁以上老年人。

尽量覆盖：

- 60-69 / 70-79 / 80+
- 独居 / 非独居
- 数字技术熟悉 / 不熟悉
- 不同职业和教育背景

前期不建议纳入严重认知障碍参与者，除非已有伦理审批和照护者协助。

### 8.3 访谈材料

完整问卷见：

`relationship-aware-interview-questionnaire.md`

主要材料包括：

1. 话题卡：工作、家庭、旧物、故人、新技术等。
2. 角色卡：同龄共鸣者、晚辈好奇者、中年传承者等。
3. 三种话题引入方式脚本：直接提问、单角色铺垫、多角色短对话。
4. 记忆卡片样例：保存、修改、不保存、以后不要再提。

### 8.4 访谈流程

| 阶段 | 内容 | 目的 |
|---|---|---|
| A 基本背景 | 年龄、居住情况、技术使用经验 | 理解参与者背景 |
| B 日常回忆习惯 | 平时和谁讲、什么时候想起过去 | 理解真实回忆场景 |
| C 话题卡排序 | 愿意聊/看情况/不想聊 | 找出话题边界 |
| D 角色卡匹配 | 每类话题适合什么角色 | 建立话题-角色矩阵 |
| E 引入方式对比 | 直接提问 vs 单角色 vs 多角色短对话 | 验证社会线索接受度 |
| F 记忆与控制 | 什么能记、什么不能记 | 建立记忆授权规则 |
| G 敏感话题边界 | 故人、悲伤、隐私、依赖 | 建立边界守护原则 |

### 8.5 访谈产出

访谈后不只是写总结，而是要产出四个可用于系统设计的结果：

1. 老年人陪伴关系角色分类。
2. 话题-关系角色匹配矩阵。
3. 社会线索引入话题的设计规则。
4. 长期记忆和敏感话题边界规则。

## 9. Study 2：原型与对比实验

Study 2 可以在 Study 1 之后进行，不需要一开始就做完整自动系统。可以先用 Wizard-of-Oz 或半自动原型。

### 9.1 实验条件

| 条件 | 形式 | 验证目的 |
|---|---|---|
| C1 Single-agent | 一个 AI 陪伴者直接提问 | 基础对照 |
| C2 Fixed-role multi-agent | 固定同龄 + 晚辈角色 | 对照 ReminiBuddy 式固定多角色 |
| C3 Relationship-aware multi-agent | 根据话题和记忆动态选择角色，并支持 agent-agent cueing | 验证关系编排和社会线索机制 |

### 9.2 任务材料

每位参与者可以使用 2-3 类材料：

- 个人照片
- 通用怀旧物件图片
- 老歌或地方文化材料
- 工作/家庭/新技术等话题卡

### 9.3 实验任务

每个条件下，参与者围绕一个材料或话题进行 5-8 分钟对话。

每轮结束后填写或口头回答：

- 哪种方式更自然？
- 哪种方式更想继续讲？
- 哪种方式压力更小？
- 是否觉得有人陪伴？
- 是否觉得 AI 太多、太吵或太假？
- 是否愿意让系统记住这段内容？

### 9.4 评价指标

内容层：

| 指标 | 说明 |
|---|---|
| 自传式记忆数量 | 是否讲到个人经历 |
| 具体细节数量 | 人物、地点、时间、事件、情绪 |
| 今昔对比 | 是否比较过去和现在 |
| 自我表达深度 | 是否出现价值判断、遗憾、骄傲、人生总结 |
| 主动补充 | 是否在 AI 问题外主动扩展 |

体验层：

| 指标 | 说明 |
|---|---|
| 陪伴感 | 是否觉得有人一起聊 |
| 社会临场感 | 是否感到多人聊天氛围 |
| 信任感 | 是否觉得 AI 理解自己 |
| 控制感 | 是否能暂停、换人、拒绝保存 |
| 开口压力 | 是否更容易开始讲 |
| 认知负担 | 是否觉得角色太多或太乱 |
| 隐私舒适度 | 是否担心被记录或冒犯 |

行为层：

| 指标 | 说明 |
|---|---|
| 用户说话时长 | 参与者讲述总时长 |
| 平均回应长度 | 每轮回应字数或时长 |
| 打断次数 | 用户是否打断 AI 或被 AI 打断 |
| 角色切换接受度 | 用户是否接受换角色 |
| 记忆确认行为 | 保存、修改、不保存、不要再提 |
| agent-agent cue 后回应 | 多角色短对话后回应是否更长、更具体 |

## 10. 原型实现范围

### 10.1 当前不必马上完整开发

细化阶段不需要马上做完整系统。可以先做低保真或 Wizard-of-Oz：

- 用固定脚本模拟三种话题引入方式
- 用人工选择角色模拟关系编排
- 用表格记录记忆卡片
- 用访谈反馈修正规则

### 10.2 后续可实现的技术架构

如果进入原型阶段，可以采用：

| 层级 | 技术 | 作用 |
|---|---|---|
| 前端 | React | 多角色聊天界面、记忆卡片、控制按钮 |
| 后端 | FastAPI | 对话接口、实验条件管理、日志保存 |
| 多 agent 编排 | LangGraph 或 AutoGen | 管理角色、状态、发言顺序和边界判断 |
| 数据库 | SQLite 或 PostgreSQL | 保存授权记忆、访谈记录、实验 trace |
| 研究后台 | 简单 dashboard | 查看 role trace、topic trace、memory trace |

### 10.3 最小可行原型

最小版本只需要实现：

1. 三种对话条件切换。
2. 2-3 个 AI 角色气泡。
3. 话题卡或图片输入。
4. “换一个人聊”“先别问了”“帮我总结”“不要记住”按钮。
5. 记忆卡片确认。
6. 研究者后台记录每轮使用的角色和话题。

## 11. 预期贡献

### 理论贡献

提出“关系感知型多智能体陪伴”的视角：老年陪伴中重要的不只是 agent 数量，而是关系角色如何随话题和用户需求动态变化。

### 机制贡献

提出 agent-agent conversational cueing：让多个 AI 先短暂互动，再邀请老人加入，把社会性对话本身变成记忆线索。

### 设计贡献

形成面向老年回忆陪伴的设计原则：

- 话题-关系角色匹配
- 低负担角色控制
- 选择性发言和沉默
- 记忆保存确认
- 敏感话题边界守护

### 方法贡献

提供从访谈到系统编排规则的流程：

> 访谈 → 话题卡/角色卡共创 → 角色需求分类 → 编排规则 → 原型对比 → 内容编码和体验评估

## 12. 可能风险与收缩方案

| 风险 | 表现 | 收缩方式 |
|---|---|---|
| 方向太大 | 同时想做陪伴、记忆、伦理、技术实现 | 主线只保留关系编排和社会线索 |
| 多 agent 太复杂 | 老人觉得吵或混乱 | 每轮最多 2-3 个角色，每个只说 1 句 |
| 伦理边界太敏感 | 故人、疾病、孤独话题可能引发不适 | 前期只作为边界问题，不主动深入 |
| 系统实现压力大 | 完整 LangGraph 原型耗时 | 先用 Wizard-of-Oz 或半自动脚本 |
| 评价指标太多 | 数据分析困难 | 先聚焦回忆细节、开口压力、陪伴感、控制感 |
| 文献跨度太广 | related work 变成堆砌 | 按三条机制组织：回忆线索、关系角色、多 agent 编排 |

## 14. 可以跟老师汇报的简短版本

老师，我理解您说的细化，是先把研究从“多智能体老年陪伴”收束到可验证的机制上。我现在把重点放在三个方面：第一是关系编排，也就是不同话题下由什么样的 AI 陪伴角色出现；第二是社会线索引入话题，也就是多个 AI 先围绕照片或旧物短暂对话，再邀请老人加入；第三是多智能体角色交互，也就是谁共鸣、谁追问、谁总结、什么时候应该停止。

我接下来会先做访谈和共创，不急着完整开发系统。访谈会用话题卡、角色卡、三种话题引入方式和记忆卡片，去了解老人对不同话题、不同角色和多个 AI 先聊天这种方式的接受度。访谈结果会转化成话题-角色匹配矩阵、社会线索设计规则和记忆边界规则。之后再做 Wizard-of-Oz 或半自动原型，对比 single-agent、fixed-role multi-agent 和 relationship-aware multi-agent 三种条件。

这样我的研究就不是简单增加 agent 数量，而是研究老年陪伴中关系角色如何被动态编排，以及 agent-agent conversation 能不能成为一种新的社会性记忆线索。
