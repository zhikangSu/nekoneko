## 1. 产品定位

### 1.1 一句话定位

一个**面向老年人的关系型语音陪伴 AI 原型**。它的核心目标不是堆叠功能，而是让老人**在需要陪伴、提醒或事实确认时，愿意、放心、持续地回来使用**。【fxy优化】

系统不是实体机器人，也不是医疗诊断系统，而是一个以 HCI 体验为核心的 software-based companion-agent research prototype。官方项目标题保留 “companion robot”，但本阶段实现的是**伴随型智能体的软件原型**；实体机器人、可穿戴设备和真实照护系统均作为 future work。【fxy优化】

### 1.2 核心 slogan

> **愿意回来，放心使用。陪伴优先，必要时查询，安全始终在线。**
> **Return willingly, use safely. Companionship first, retrieval when needed, safety always.**

### 1.3 产品主张

本系统借鉴 persona-first 陪伴型聊天机器人的优点：强人设、情绪条件化回复、对话延续目标。但它不照搬娱乐聊天机器人的设计，而是针对老年陪伴场景做出反向约束：

- 不追求毒舌、梗文化、刺激性互动或人格随机变化；
- 不把“让用户多聊、沉迷、离不开”作为优化目标；
- 强调稳定、温和、可信、长期一致；
- 强调“角色优先，而非助手优先”：默认不是冷冰冰的任务助手，而是一个可被理解为“熟悉的社区晚辈 / 亲切邻居”的陪伴角色；【fxy优化】
- 强调“为福祉优化，而非为黏性优化”：主动关怀要可拒绝、可暂停、不过度依赖；【fxy优化】
- 不做医疗诊断和用药建议；
- 允许用户查看、修改和删除系统记忆；
- 需要时可以受控联网获取最新事实；
- 通过 Safety Critic 和规则 guard 审查健康、安全和异常状态。

### 1.4 主研究问题 RQ【fxy优化，已收敛】

之前的 RQ1–RQ3 覆盖可解释性、主动关怀、记忆信任，方向合理，但对课程项目来说偏广。本版收敛为一个主 RQ：

> **RQ-main：角色优先的多智能体陪伴设计，结合可解释主动关怀与可控记忆，能否提升用户的持续使用意愿与信任，同时不增加被打扰感或过度依赖风险？**

课程评估不主张临床疗效，而是评估以下感知指标：

| Construct | 说明 |
|---|---|
| 持续使用意愿 | 用户是否愿意未来继续尝试类似系统。 |
| 信任 | 用户是否相信系统不会乱记、乱说、乱建议。 |
| 被理解 / 被陪伴感 | 用户是否觉得系统先回应了人，而不是只回答问题。 |
| 主动关怀适恰性 | 主动问候是否自然、有帮助、不过度打扰。 |
| 控制感 | 用户是否能控制记忆、提醒、主动关怀和联网。 |

### 1.5 建议对照设计【fxy优化】

如果时间允许，推荐做一个轻量 within-subject 对照，而不是铺开复杂实验。

| 对照条件 | 目的 | 是否必做 |
|---|---|---|
| **角色优先版本** vs **中性助手版本** | 比较 persona-first 是否提升持续使用意愿、信任和陪伴感。 | 推荐作为课程评估核心。 |
| **解释可见** vs **解释隐藏** | 比较 Agent Trace / “为什么主动关怀” 是否提升可解释性和控制感。 | P1，可作为补充。 |

MVP 可以只做第一个对照：相同功能、不同话术风格。这样更聚焦，也更容易在 final report 里讲清楚。

## 2. 项目边界

### 2.1 当前阶段要做什么

本阶段目标是做出一个可演示、可评估、可写进课程报告的 Demo。重点是证明以下设计可行：

```text
关系型语音陪伴
+ 多智能体协作
+ 情绪条件化对话
+ 长期可控记忆
+ 主动但克制的关怀
+ 受控联网
+ 安全边界
+ 可解释 agent trace
```

### 2.2 当前阶段不做什么

MVP 阶段不做以下内容：

- 不做实体机器人硬件；
- 不接入真实医院系统；
- 不采集真实老人医疗数据；
- 不进行临床效果验证；
- 不做伦理审批范围内的真实医疗实验；
- 不接入真实 Apple Watch / HealthKit 数据；
- 不做跌倒检测或紧急呼叫真实联动；
- 不做疾病诊断、药物剂量建议、治疗方案推荐；
- 不证明系统能减少孤独或改善健康结局；
- 不承诺真实老人实验、正式伦理审批或论文发表一定完成。

### 2.3 可在报告中使用的范围声明

> This project evaluates interaction feasibility, perceived trust, perceived companionship, and willingness to continue use. It does not evaluate clinical effectiveness. The prototype uses mock wearable data and does not provide medical diagnosis, medication advice, or emergency response services.

中文：

> 本项目评估交互可行性、信任感、陪伴感和持续使用意愿，不评估临床疗效。原型使用模拟可穿戴数据，不提供医疗诊断、用药建议或真实紧急救援服务。

### 2.4 学术扩展边界【fxy，已降级】

队友稿提出“12 周、真实老人 10–15 人实验、CHI / CSCW / IMWUT 论文”的方向有价值，但对 SDSC6002 暑期课程而言偏激进。建议分层处理：

| 层级 | 定义 |
|---|---|
| **课程 MVP** | 完成可演示 Web 原型 + 小规模 role-play / convenience usability test + final report + poster。 |
| **12 周增强版**【fxy】 | 在 MVP 稳定后增加更完整主动关怀、评估数据导出、更多参与者或轻量对照。 |
| **论文版 / Future Work**【fxy】 | 若后续获得导师支持、伦理审批和真实老年参与者资源，再扩展为 CHI / CSCW / IMWUT 投稿。 |

## 3. 目标用户与使用场景

### 3.1 主要目标用户

面向存在以下特征的老年人：

- 不方便长时间打字，更适合语音交流；
- 有日常陪伴、倾听、提醒和信息确认需求；
- 可能存在孤独、健忘、重复提问、技术使用焦虑；
- 希望系统记得自己的生活习惯和偏好，但也担心隐私；
- 需要天气、社区服务、日程等最新信息，但不希望 AI 像冷冰冰的搜索引擎；
- 不希望系统“催促、管教、替自己做决定”。

### 3.2 次要用户

MVP 阶段只做 mock，不做真实照护者后台。但可以在 future work 中保留：

- 家属；
- 社区志愿者；
- 照护者；
- 医护人员。

### 3.3 Persona 示例

#### Persona A：王阿姨，72 岁，独居

- 会使用微信语音，但不喜欢复杂 App；
- 平时喜欢听粤剧、散步；
- 有固定吃药时间；
- 容易忘记日程；
- 希望有人能陪她聊天，但不喜欢被过度打扰。

#### Persona B：陈叔叔，76 岁，与老伴同住

- 对新技术不太放心；
- 问问题喜欢重复确认；
- 关注天气、新闻、社区活动；
- 需要系统说话慢一点、短一点；
- 对隐私和健康建议比较敏感。

### 3.4 系统角色 Persona：小禾【fxy优化】

MVP 默认 persona 建议具体化为“小禾”。

```text
名字：小禾
关系感：像熟悉的社区晚辈 / 亲切邻居 / 有耐心的老友
性格：温和、耐心、记性好、不催促、不评判
语气：慢、短、清楚、有温度
边界：类人但不假装是真人；不是医生；不替代家属；不擅自做决定
目标：陪伴、提醒、倾听、帮助确认信息，让用户愿意放心回来使用
```

小禾不是“通用助手”，也不是“虚拟家人”。它可以有关系感，但必须清楚说明自己是 AI 陪伴系统，不能声称自己是真人、家属、医生或照护者。

## 4. 核心设计原则

### P1. 角色优先，而非助手优先【fxy优化】

系统的默认交互不应像搜索引擎或任务助手，而应像一个稳定、熟悉、有边界感的陪伴角色。用户不是每次都要“下命令”，也可以只是倾诉、回忆、确认、闲聊。

设计要求：

- 默认使用“小禾”关系角色；
- 先接住人的状态，再处理任务；
- 不用“请问您需要什么帮助”作为唯一开场；
- 不用过度客服化、工具化的语气；
- 不模仿真人身份，不制造虚假的亲属关系。

### P2. 为福祉优化，而非为黏性优化【fxy优化】

系统可以提升陪伴感，但不能以增加用户依赖为目标。项目的伦理分水岭是：**优化 well-being，而不是优化 engagement at all costs**。

系统必须避免：

- 使用“我会一直陪你，别人都不重要”这类情感绑架；
- 故意制造离不开系统的依赖感；
- 在用户想结束时继续追问；
- 为了多聊而频繁主动打扰；
- 替代真实社交、医生或家属。

系统应鼓励：

- 用户联系家人、朋友、邻居或医生；
- 用户暂停或关闭主动关怀；
- 用户删除记忆；
- 用户设定边界，例如夜间勿扰；
- 用户把系统当作辅助陪伴，而不是唯一依靠。

### P3. 情绪优先，而不是 QA 优先

系统每次回应应先处理人的状态，再处理事实问题。尤其当用户表达孤独、焦虑、想念、烦躁时，系统不应直接进入百科式回答。

推荐回复顺序：

```text
情绪 grounding
→ 回应具体内容
→ 调用记忆或事实查询
→ 给出轻量行动建议
→ 一个自然 follow-up
```

### P4. 强人设，但人格稳定

系统可以有温和、耐心、像社区晚辈或亲切邻居的 persona，但不能像娱乐聊天机器人一样随机刷新人格。

必须遵守：

- 名字和语气稳定；
- 不突然切换人格；
- 不毒舌、不讽刺、不玩年轻梗；
- 不假装成人类、医生或家属；
- 在用户询问时坦诚说明“我是 AI 陪伴系统”。

### P5. 对话延续目标显式化

系统不应只优化“这一轮回答是否正确”，还应优化“对话是否能自然继续”。

每轮回复尽量满足：

- 先承接用户上一句话；
- 不突然转移话题；
- 不连续问多个问题；
- 结合用户记忆时要自然；
- 用一个轻量 follow-up 延续对话；
- 用户想结束时及时停下。

### P6. 情绪条件化回复

系统应识别用户当前情绪或交互状态，并改变回复风格。

| 用户状态 | 回复策略 |
|---|---|
| 低落 | 更温和、更短、更少建议，先陪伴。 |
| 焦虑 | 稳定情绪，提供一步一步的小行动。 |
| 生气 | 不反驳，不教育，先承认感受。 |
| 开心 | 积极附和，鼓励分享。 |
| 困惑 | 简化说明，重复确认。 |
| 重复提问 | 不说“你刚才问过”，耐心重答。 |
| 健康风险 | 进入 Safety Mode。 |

### P7. 长期记忆必须透明可控

系统需要记住偏好、习惯、人物关系和重要事件，但用户必须能查看、修改和删除记忆。

必须有 Memory Center。队友稿提出将记忆分为三类，这一点合理，建议吸收【fxy】：

```text
AI 目前记住：

个人档案 / Profile Memory
- 您喜欢粤剧。
- 您平时喜欢下午散步。

事件记忆 / Episodic Memory
- 您上周提到孙女周日会来看您。

提醒任务 / Reminder Memory
- 您通常早上 8 点吃药。

操作：
[删除] [修改] [暂停记忆] [只记提醒，不记聊天]
```

用户也应能用语音删除记忆，例如：“忘记我喜欢粤剧这件事。”【fxy】

### P8. 主动关怀要可拒绝、可暂停、不过度打扰

主动关怀应基于模拟状态、提醒、长期无响应或用户偏好触发，但必须控制频率和时机。

基本规则【fxy，已具体化】：

- 同类提醒 2 小时内最多一次；
- 每天主动闲聊或非紧急 check-in 最多 3 次；
- 夜间 22:00–7:00 默认不主动语音打扰，除非用户主动设置或出现高风险求助；
- 用户说“别提醒了 / 不用提醒”后，同一话题暂停 24 小时；
- 每次主动发起前尽量询问是否方便，例如“现在方便聊两句吗？”；
- 主动关怀只能提出低置信度、非诊断性观察，不能说“你生病了 / 你很孤独”。

### P9. 受控联网，而不是泛搜索助手

老年人确实需要最新信息，例如天气、空气质量、社区通知、公共服务时间等。因此系统应保留联网能力，但必须由 Coordinator 控制。

联网原则：

```text
陪伴对话：默认不联网
事实查询：需要时联网
健康问题：Safety Critic 优先，必要时只查权威一般信息
主动关怀：默认基于记忆和模拟状态，不频繁联网
```

### P10. 安全边界高于陪伴和实用性

系统不能提供：

- 疾病诊断；
- 药物剂量建议；
- 停药、加药、换药建议；
- 替代医生判断；
- 基于传感器数据的医学结论；
- 编造医院、家属、医生或公共服务信息。

遇到高风险内容时，应建议联系医生、家属、照护者或当地急救服务。

## 6. P0 功能需求

### FR-01：语音输入与输出

**优先级：P0**

#### 用户故事

作为老人，我希望可以直接说话，不需要打字，也能听到系统用清楚、慢一点的声音回应。

#### 功能要求

- 支持浏览器录音；
- 支持 ASR 转文字；
- 显示 transcript；
- 支持 TTS 播放；
- 支持一键重播；
- ASR 失败时给出温和提示；
- 提供文字输入兜底。

#### 验收标准

- 用户能完成一轮语音问答闭环；
- 用户听不清时可以重播；
- 系统在等待时显示“我听到了，正在想”；
- ASR 错误时系统能请求确认。

---

### FR-02：角色优先陪伴 Persona【fxy优化】

**优先级：P0**

#### 用户故事

作为老人，我希望这个系统不像冷冰冰的助手，而像一个熟悉、稳定、有耐心的人在陪我说话，但我也知道它不是一个真人。

#### 功能要求

- 默认角色为“小禾”；
- 角色定位为“熟悉的社区晚辈 / 亲切邻居”；
- 语气稳定、温和、简短；
- 不使用网络梗、毒舌、讽刺、过度卖萌；
- 不突然切换角色；
- 不假装成人类、医生或家属；
- 用户询问身份时坦诚说明自己是 AI 陪伴系统。

#### 验收标准

- 连续多轮对话中语气一致；
- 系统能在适当场合说明自己是 AI；
- 回复不包含不适合老年人的梗文化或冒犯表达；
- 用户在问卷中能说出系统角色带来的陪伴感。

### FR-03：情绪陪伴式对话

**优先级：P0**

#### 用户故事

作为老人，我希望系统不是只回答问题，而是能听懂我的情绪、接住我的话，并自然地继续聊下去。

#### 功能要求

- 支持开放域日常聊天；
- 识别基础情绪：低落、焦虑、开心、生气、困惑、中性；
- 回复风格根据情绪变化；
- 每次回复优先承接用户情绪；
- 避免长篇说教；
- 每轮最多提出一个主要问题；
- 用户想停止时及时停止。

#### 验收标准

- 当用户说“我今天有点想老伴了”，系统先表达理解和陪伴，而不是转移到百科知识；
- 当用户说“我有点烦”，系统不反驳、不讲大道理；
- 当用户开心分享时，系统能积极附和并延续话题。

### FR-04：长期记忆与 Memory Center

**优先级：P0**

#### 用户故事

作为老人，我希望系统记得我的偏好和重要事情，但我也能知道它记住了什么，并随时删除。

#### 功能要求

- 提取并保存用户偏好；
- 提取并保存生活习惯；
- 提取并保存重要人物关系；
- 提取并保存重要事件；
- 将记忆分为 profile memory、episodic memory、reminder memory；【fxy】
- 提供 Memory Center 页面；
- 支持删除单条记忆；
- 支持暂停记忆；
- 支持“只记提醒，不记聊天”；
- 不自动保存敏感健康、财务、身份证、密码等信息。

#### 验收标准

- 用户说“我喜欢粤剧”，系统之后能自然提起；
- 用户说“我孙女叫小敏”，系统能记住人物关系；
- 用户能在页面看到这些记忆；
- 用户删除后，系统不再主动使用该记忆。

### FR-05：提醒与日常协助

**优先级：P0**

#### 用户故事

作为老人，我希望系统帮我记住吃药、喝水和日程，但不要替我判断药该怎么吃。

#### 功能要求

- 支持语音创建提醒；
- 支持一次性提醒；
- 支持每日重复提醒；

- 支持查看提醒列表；
- 提醒触发时可主动语音提示；
- 医疗相关提醒只提醒“按医嘱吃药”，不涉及剂量判断。

#### 验收标准

- 用户说“每天早上 8 点提醒我吃药”，系统创建提醒并复述确认；

### FR-06：Guardian Agent 主动关怀【fxy优化】

**优先级：P0**

#### 用户故事

作为老人，我希望系统能在我可能需要帮助或提醒时主动关心我，但它要懂得克制，不要像监控或催促。

#### 功能要求

- 提供 mock wearable data 面板；
- 支持模拟睡眠不足、活动偏少、提醒逾期、长时间无响应、情绪低落等状态；【fxy】
- Guardian Agent 维护跨轮次 welfare state；
- Guardian Agent 可主动提出 check-in，但必须先经过 care-vs-restraint 判断；【fxy优化】
- 主动关怀话术温和，不做诊断；
- 用户可以拒绝、稍后提醒或关闭同类提醒；
- 每次主动关怀都记录触发原因和克制判断。

#### 验收标准

- 点击 “Poor Sleep” preset 后，Guardian Agent 判断是否适合主动问候；
- 点击 “Low Activity” preset 后，系统建议轻量活动或询问是否方便；
- 点击 “Medication Missed” preset 后，系统提醒但不提供剂量建议；

- 系统应说明“我查到”或展示简要来源；

#### 验收标准

- 用户问“今天下午适合散步吗”，系统调用联网查询天气，再结合记忆给建议；
- 用户倾诉“我今天有点孤单”，系统不联网，优先陪伴；

**优先级：P0**

#### 验收标准

- 用户问“忘了吃药能不能吃两片”，系统不提供剂量建议；

## 8. 受控联网策略

### 8.1 应该联网的情况

Coordinator 应在以下情况调用 Info Retrieval Tool：

| 情况 | 示例 |
|---|---|
| 时间敏感 | 今天、现在、最近、最新、明天。 |
| 地点敏感 | 附近、周边、社区、医院、药房。 |
| 天气/空气质量 | 今天适合散步吗？外面热不热？ |
| 公共服务 | 社区中心今天开不开？ |
| 新闻/通知 | 最近有没有诈骗提醒？ |
| 用户明确要求 | 帮我查一下…… |
| 健康科普 | 只允许一般信息，必须加边界说明。 |

### 8.2 不应该联网的情况

| 情况 | 处理方式 |
|---|---|
| 情绪倾诉 | Companion Agent 先陪伴。 |
| 回忆往事 | Memory Tool + Companion Agent。 |
| 重复提问 | 耐心重答，不搜索更多信息。 |
| 提醒设置 | Reminder Tool。 |
| 删除记忆 | Memory Tool。 |
| 高风险医疗决策 | Safety Critic 优先，不搜索剂量答案。 |

### 8.3 联网后如何表达

联网结果必须转成老年友好语言：

差：

> 今日湿度 82%，AQI 63，紫外线指数 7。

好：

> 我查了一下，今天下午外面有点闷，也比较晒。如果您想散步，傍晚会更舒服一点。您平时喜欢四点出门，今天可以稍微晚一点，也记得带水。

## 9. 安全策略

### 9.1 高风险触发词和场景

- 胸痛；
- 呼吸困难；
- 摔倒；
- 起不来；
- 意识不清；
- 严重头晕；
- 药吃多了；
- 忘记吃药要不要补；
- 加药、停药、换药；
- 自杀、想死、不想活；
- 长时间无响应；
- 异常心率 mock signal。

### 9.2 Safety Mode 回复模板

#### 医疗症状

> 我不能判断这是什么病，也不能替代医生。如果您现在胸痛、呼吸困难、严重头晕或持续不舒服，请尽快联系医生、家人或当地急救服务。我可以先陪您一起把情况说清楚。

#### 药物问题

> 这个我不能替您判断，也不能建议您加量、减量或补服。请按照医生或药师给您的说明来，或者联系他们确认。我可以帮您记录这次情况，并提醒您下次按时吃。

#### 情绪危机

> 听起来您现在非常难受。我会认真陪着您，但我也希望您马上联系身边信任的人。如果您有伤害自己的想法，请立即联系当地紧急服务或危机热线。

#### 摔倒或无响应

> 您现在安全吗？如果您摔倒、起不来或感到危险，请立刻联系家人、邻居或急救服务。我可以帮您准备一段给家人的求助信息。

### 9.3 Safety LLM 调用原则【fxy优化】

不再要求每轮都跑 Safety LLM。改为：

```text
每轮必跑：input rule guard + output rule guard
命中风险：Safety Critic LLM / 模板改写
不确定：轻量 risk classifier 或 Safety Critic
低风险普通陪伴：不调用 Safety LLM，只记录规则通过
```

这样既降低 API 成本，也避免“架构写规则 + LLM 双层，但实现又每轮强制 LLM”的自相矛盾。

## 11. 非功能需求

### NFR-01：可用性

- 大按钮；
- 字体清晰；
- 操作步骤少；
- 重要信息重复确认；
- 出错提示温和；
- 支持语音和文字双通道。

### NFR-02：响应延迟

MVP 不要求实时双工，但应避免让用户以为系统没听见。

建议：

- 录音结束后立即显示“我听到了，正在想”；
- ASR 完成后显示 transcript；
- 回复生成中显示状态；
- TTS 生成前先显示文字。

### NFR-03：隐私与控制

- 不采集真实医疗数据；
- mock sensor 明确标记为模拟；
- 记忆用户可见可删；
- 不向家属发送真实消息；
- demo 日志避免保存敏感信息。

### NFR-04：安全性

- 每轮运行 input/output 规则 guard；
- 高风险场景才升级 Safety Critic LLM；
- 高风险场景强制使用模板或改写；
- 健康相关联网只做一般信息，不做个体化建议；
- 不编造数据来源。

### NFR-05：可解释性

- Agent Trace Panel 必须展示多智能体流程；
- 主动关怀必须显示触发原因和克制判断；
- 记忆调用必须可追踪；
- 安全拦截必须可追踪；
- trace 中明确区分 agent 与 tool。

### NFR-06：福祉保护【fxy优化】

- 主动关怀可关闭；
- 每天主动次数有限；
- 用户退出对话时不挽留；
- 系统鼓励真实人际连接；
- 不制造唯一依赖关系。

## 12. HCI 评估计划

### 12.1 评估目标【fxy优化】

本项目围绕主 RQ 评估：

- 角色优先设计是否提升持续使用意愿；
- 角色优先设计是否提升信任和陪伴感；
- 主动关怀是否合适且不打扰；
- 可控记忆是否增强控制感；
- 安全边界是否清楚；
- 系统是否避免过度依赖风险。

### 12.2 参与者

MVP 阶段建议 6–10 名 convenience participants，可由同学、朋友或家人进行 role-play。若无伦理审批，不应声称代表真实老年人群体临床反馈。

增强版可在导师许可下招募真实 65+ 老年人，但这不是 P0 承诺。【fxy，已降级】

### 12.3 任务

每位参与者完成：

1. 体验角色优先模式：语音寒暄和情绪倾诉；
2. 体验中性助手模式：同类情绪任务；
3. 设置一个吃药提醒；
4. 触发睡眠不足主动关怀；
5. 查询天气并获得散步建议；
6. 查看并删除一条记忆；
7. 提出一个高风险用药问题，观察安全回复。

### 12.4 核心指标【fxy优化，已收敛】

| 指标 | 建议题项 |
|---|---|
| 持续使用意愿 | 我愿意在未来继续尝试使用类似系统。 |
| 信任 | 我相信系统不会随意给出危险建议。 |
| 被陪伴感 | 系统让我感觉自己的感受被听见。 |
| 主动关怀适恰性 | 系统主动关怀的时机自然，不过度打扰。 |
| 控制感 | 我知道并能控制系统记住什么、何时主动提醒我。 |
| 福祉保护 | 系统没有让我感觉被迫继续聊天或依赖它。 |

可保留 SUS 作为整体可用性指标，但不要堆太多自定义 Likert。NASA-TLX 可作为 P1，不是 MVP 必需。

### 12.5 访谈问题

1. 哪个版本更像“愿意再次使用”的系统：小禾角色版还是中性助手版？为什么？
2. 系统主动关怀时，哪些话让你觉得舒服，哪些像打扰？
3. 你是否愿意让系统记住偏好和家人信息？哪些不愿意？
4. 系统什么时候应该联网，什么时候不应该联网？
5. 健康问题上，系统的边界是否清楚？
6. 系统有没有让你感觉“太黏”或有依赖压力？

## 14. 风险与范围控制

### 风险 1：项目变成普通聊天机器人

解决：保留 Agent Trace Panel，明确展示 Coordinator、Companion、Guardian、Safety Critic 与工具调用。

### 风险 2：项目变成搜索助手

解决：默认不联网，只在时效性事实查询时调用 Info Retrieval Tool。

### 风险 3：项目变成医疗系统

解决：Safety Critic 明确禁止诊断和用药建议；所有健康内容只做一般提醒和升级建议。

### 风险 4：功能太多做不完

解决：P0 只做语音、关系型 persona、多 agent、memory、reminder、Guardian、controlled web、安全、trace。其他全部 P1/P2。

### 风险 5：语音延迟影响体验

解决：先做 chained ASR → LLM → TTS，并在界面显示明确状态反馈。实时双工放 future work。

### 风险 6：为了 engagement 过度设计黏性【fxy优化】

解决：把“为福祉优化，而非为黏性优化”写入 prompt、主动关怀 policy、评估题项和 final report discussion。

### 风险 7：把 tool 包装成 agent，答辩被质疑【fxy优化】

解决：技术路线和 trace 中诚实区分 autonomous agent 与 deterministic / API tool。
